1
How would you design a scalable system / data warehouse to handle millions of requests per day?
ArchitecturePattern
▼

A scalable data system is built in layers, each with a specific role. The standard approach in Azure is the Medallion Lakehouse architecture:

  • Ingestion Layer: ADF, Event Hubs, or Kafka pull data from diverse sources. The goal is to decouple sources from processing so either side can change independently.
  • Bronze (Raw) Layer: Data lands as-is in ADLS Gen2 — no transformation. This is your immutable source of truth and replay foundation.
  • Silver Layer: Data is cleaned, validated, deduplicated, and joined. Runs in Databricks or Synapse Spark, written in Delta format for ACID compliance.
  • Gold Layer: Business-aggregated, query-ready tables served via Synapse Dedicated Pool or Serverless SQL for BI tools like Power BI.
  • Orchestration: ADF or Synapse Pipelines trigger each stage with retry logic, dependency management, and monitoring built in.
  • Scaling: Databricks auto-scales clusters; Synapse DWUs scale on demand; ADLS is inherently elastic — pay for what you use.
2
What are the differences between horizontal and vertical scaling? When would you use each?
Fundamentals
▼

Vertical scaling (Scale Up) means adding more CPU, RAM, or disk to the same machine. Simpler — no code changes — but has a hard ceiling and creates a single point of failure.

Horizontal scaling (Scale Out) means adding more machines or nodes. Requires distributed systems thinking but offers virtually unlimited ceiling.

  • Use vertical scaling for relational DBs that are hard to shard, or for quick wins on legacy systems.
  • Use horizontal scaling for Spark jobs, Kafka brokers, ADLS workloads — anywhere data partitions well across nodes.

How would you design a cost-optimized batch pipeline in Azure for large-scale daily processing?
Cost Optimization
▼
  • Spot/low-priority Databricks clusters — up to 90% cheaper for non-time-critical batch jobs. Add retry policies since spot nodes can be evicted.
  • Pause Synapse Dedicated Pool when not in use — you only pay for DWUs when the pool is running. Use ADF schedules to auto-pause post-job.
  • Process only incremental data using watermarks or CDC — avoid full-table scans every night.
  • ADLS lifecycle policies to automatically tier cold data to Cool or Archive storage.
  • Compress output files using Parquet + Snappy to reduce storage and I/O costs.
  • Right-size clusters: profile job memory and CPU, pick the smallest cluster that fits.
  • Azure Reserved Instances for always-on workloads — commit 1-3 years for up to 40% savings.

How would you process incremental data without creating duplicates (Idempotency)?
Pattern
▼

An idempotent pipeline produces the same result whether it runs once or ten times. This is critical because pipelines fail and need to re-run.

  • Watermark table: Store the last processed timestamp or ID. Each run reads only rows newer than the watermark. After a successful load, update the watermark.
  • Delta Lake MERGE (UPSERT): Match on a unique business key — update if the row exists, insert if new. Naturally handles duplicates from re-runs.
  • Partition overwrite: Write new data to a date partition, overwrite that partition on re-run. Avoids duplicates by design when partitioned by ingestion date.
  • Dedup in PySpark: df.dropDuplicates(["id", "updated_at"]) before writing.

A pipeline completed successfully but produced incorrect data. How would you debug it?
Debugging
▼

This is the hardest class of bug — the system thinks everything succeeded. Work through these steps:

  1. Narrow down the layer. Check Bronze → Silver → Gold sequentially. At which layer does data first look wrong?
  2. Check transformation logic. Are joins on the wrong key? Is a GROUP BY missing a column? Are window functions using the right partition/order?
  3. Check data quality at source. Was the source data itself corrupt, late-arriving, or duplicated?
  4. Check watermark/filter logic. Did the incremental filter cut off valid records or include old ones twice?
  5. Check date/timezone issues. UTC vs local time bugs are extremely common.
  6. Enable row counts + checksums. Add audit columns at each layer — row_count, checksum, load_timestamp — so discrepancies become visible.
  7. Delta Lake time travel: SELECT * FROM table VERSION AS OF N to compare data before and after a specific run.

A data pipeline is failing intermittently in production. How would you debug and fix it?
Debugging
▼

Intermittent failures do not always reproduce. Think in categories:

  • Infrastructure/transient: Network timeouts, API rate limits, throttling from source. Fix: retry policies with exponential backoff.
  • Data-driven: A bad record arrives on some runs (null in a non-nullable field, unexpected schema). Fix: schema validation + dead-letter queues.
  • Resource contention: Cluster runs OOM on days when data volume spikes. Fix: monitor run sizes, auto-scale, or increase shuffle partitions dynamically.
  • Concurrency: Two pipeline runs overlap and conflict on a shared table. Fix: set pipeline concurrency to 1 or use table-level locks.
🛠
Tools: ADF Monitor → Pipeline Runs → Activity runs. Databricks Job Run history for Spark logs. Azure Monitor + Log Analytics for pattern analysis across runs.

1
Under what networking conditions must you use a Self-Hosted Integration Runtime (SHIR) instead of Azure IR?
Networking
▼

The Integration Runtime (IR) is the compute engine that moves data in ADF. Azure IR is Microsoft-managed, lives in the cloud, and works for cloud-to-cloud transfers. It requires sources reachable over the public internet.

Use Self-Hosted IR (SHIR) — a Windows agent installed on a machine in your own network — when:

  • Data source is on-premises (SQL Server, Oracle, SAP) behind a corporate firewall.
  • Source is in a private VNet with no public endpoint and no peering to ADF's managed VNet.
  • You need to copy data between two private networks.
  • Source requires a local driver (ODBC, Oracle client) installed on the IR machine.
✅
The SHIR machine initiates outbound connections to ADF's cloud service over port 443 — no inbound firewall rules required.

How do you implement a Watermark-based incremental load in ADF to avoid re-processing old data?
Pattern
▼

A watermark is a "last processed" marker — usually a timestamp or integer ID. Each run fetches only records newer than this marker.

  1. Create a watermark table in Azure SQL: CREATE TABLE watermark (table_name VARCHAR, last_value DATETIME)
  2. Lookup Activity — read the current watermark value for the target table.
  3. Lookup Activity — read the current MAX value in source: SELECT MAX(updated_at) FROM orders
  4. Copy Activity — source query uses both values: WHERE updated_at > old_wm AND updated_at <= new_max
  5. Stored Procedure Activity — after successful copy, update the watermark to the new max value.
⚠️
Capturing the new max before the copy starts prevents late-arriving records from being missed on subsequent runs.

What is the difference between Copy Activity and Data Flow in ADF?
Core Concept
▼

Copy Activity is a lightweight, high-speed data movement tool. It moves data from source to sink with minimal transformation — column renaming, basic type casting. It does not spin up a Spark cluster; it runs on the Integration Runtime directly. Very fast, very cheap. Use it for ingestion — landing raw files in ADLS.

Data Flow (Mapping Data Flow) is a visual, code-free transformation engine backed by Apache Spark. You can join, aggregate, pivot, derive columns, parse JSON, and split streams — all via a drag-and-drop UI. ADF compiles your Data Flow into Spark code and runs it on an auto-created cluster.

💸
Data Flow is significantly more expensive — it spins up Spark compute. Only use it when transformation complexity justifies it. For simple copy-and-land, always use Copy Activity.

How do you implement parameterization in ADF pipelines?
Best Practice
▼
  • Pipeline parameters: Defined at pipeline level. Passed when triggering via schedule trigger, event trigger, or Execute Pipeline activity.
  • Dataset parameters: Make dataset properties dynamic — table name, file path, container name. Reference with @dataset().paramName.
  • Linked Service parameters: Parameterize connection strings so one linked service handles multiple environments (dev/prod).
  • Global parameters: Set once at ADF level, available across all pipelines — useful for environment-level config like storage account names.
  • ForEach + parameterized pipeline: Build a config-driven framework — store table names in a control config, loop with ForEach, call a generic pipeline per table with the table name as a parameter.
5
Describe how error handling and retry mechanisms work in ADF.
Reliability
▼
  • Activity retry: Each activity has configurable retry count (default 0) and retry interval. ADF re-runs the failed activity N times before marking it failed. Use for transient failures like network blips and throttling.
  • On Failure path: Each activity has Success, Failure, and Completion output paths. Connect activities on the Failure path to trigger cleanup, send alerts (Web Activity to Teams webhook), or log errors to a SQL table.
  • Timeout: Set timeouts on long-running activities to prevent hanging pipelines from consuming resources indefinitely.
  • Fault tolerance in Copy Activity: Skip incompatible rows and log them to a fault file in ADLS instead of failing the entire copy. Configure under Copy Activity → Settings → Fault tolerance.

How would you securely migrate on-premises SQL Server data into ADLS Gen2?
SecurityMigration
▼
  1. Install SHIR on a machine in the on-prem network that can reach the SQL Server.
  2. Create a Linked Service in ADF pointing to on-prem SQL Server via the SHIR. Store the password in Azure Key Vault — never hardcode credentials.
  3. Use Managed Identity for ADF to authenticate to ADLS Gen2 — no storage account keys needed.
  4. Configure a Copy Activity with the on-prem SQL linked service as source and ADLS Gen2 as sink. Use Parquet format for storage efficiency.
  5. Ensure encryption in transit: SQL Server uses TLS, ADLS uses HTTPS, SHIR to ADF cloud uses TLS over port 443.
  6. Implement incremental load using watermarks or SQL Server Change Tracking to avoid moving the full dataset on every run.
  7. For high-security requirements, use ExpressRoute or VPN for private connectivity instead of public internet traversal.

ADLS Gen2 vs Azure Blob Storage — what are the key differences?
Storage
▼

Both are Azure object storage, but ADLS Gen2 = Blob Storage with Hierarchical Namespace (HNS) enabled. That one feature creates substantial differences:

  • Namespace: Blob = flat (folders are just key prefixes, no real directories). ADLS Gen2 = true hierarchical file system — directories are first-class objects. Rename a "folder" of 1 million files: Blob O(n), ADLS Gen2 O(1).
  • Atomic operations: ADLS Gen2 supports atomic rename and delete at directory level — critical for Spark job commits. Blob Storage requires listing and updating every blob, which is non-atomic and slow.
  • POSIX ACLs: ADLS Gen2 supports POSIX-style ACLs at file and directory level (read/write/execute per user/group). Blob Storage only has RBAC at container level.
  • Performance: ADLS Gen2 is optimized for big data analytics workloads (Hadoop/Spark compatible). Significantly higher throughput for parallel reads/writes.
  • Protocol: ADLS Gen2 supports both the DFS endpoint (abfss://) for analytics and the Blob endpoint (wasbs://) for backward compatibility.
  • Cost: ADLS Gen2 costs slightly more due to namespace metadata storage, but the performance gains far outweigh this for analytics.
📌
Rule: Use Blob Storage for simple file hosting, CDN origin, or backup. Use ADLS Gen2 for any analytics workload — Spark, ADF, Databricks, Synapse. Always enable HNS at account creation; it cannot be added later.

How do you handle schema drift in ADF?
Schema Drift
▼

Schema drift happens when source systems add, remove, or rename columns without telling you. In ADF, you handle it at two levels:

In Mapping Data Flow (the right place to handle drift):

  • Enable "Allow schema drift" on the Source transformation — ADF will accept any columns from the source, even ones not in the schema definition.
  • Enable "Infer drifted column types" — ADF will auto-detect the data type of new columns.
  • Use "Auto mapping" on the Sink transformation — ADF automatically maps source columns to sink, including new drifted ones.
  • Use drifted column patterns (regex-based column selectors) in Derived Column transforms to apply logic to all columns matching a pattern, e.g., all columns ending in _id.

In Copy Activity: Copy Activity doesn't transform, so schema drift at copy time is handled by writing raw data as-is (landing in ADLS as JSON/Parquet with all columns). Downstream Databricks handles the schema reconciliation.

Best practice for production: Don't silently accept all drift. Add a schema validation step that compares incoming schema to expected schema — alert on changes so the team can decide consciously whether to accept or reject them.

How do you manage Linked Services and Datasets in ADF at scale?
Best Practice
▼
  • Parameterize everything: A single linked service per technology (one for Azure SQL, one for ADLS), parameterized with server, database, and container name. Datasets also parameterized with table name, file path, schema. This means one linked service handles 100 tables, not 100 linked services.
  • Store credentials in Key Vault: Linked services should reference Key Vault secrets, never hardcode connection strings. Use Managed Identity for Azure resources (ADLS, Azure SQL) — no secrets at all.
  • Use Integration Runtime appropriately: One SHIR per on-prem network zone. Azure IR for all cloud-to-cloud. Don't create a new IR for every pipeline.
  • Naming convention: ls_[source]_[env] (e.g., ls_sqlserver_prod), ds_[source]_[format] (e.g., ds_adls_parquet). Consistent naming makes maintenance at scale manageable.
  • Git integration: Store all ADF artifacts (pipelines, linked services, datasets, triggers) in Azure DevOps or GitHub. Use ADF's native Git integration. Never edit directly in Production — always go through dev → test → prod via CI/CD.
  • ARM template deployment: ADF publishes a ARMTemplateForFactory.json that CI/CD pipelines use to deploy across environments, with environment-specific parameter files replacing dev values with prod values.

How do you deploy ADF across environments (Dev → Test → Prod)?
CI/CD
▼
  1. Connect ADF to Git (Azure DevOps or GitHub): All changes made in ADF Studio are committed to a feature branch. Dev ADF is the only one connected to Git.
  2. Collaboration branch: PRs merge into main/adf_publish. ADF auto-generates ARM templates on publish.
  3. CI Pipeline (Azure DevOps): On merge to main, validate ARM templates, run unit tests on pipeline logic if applicable.
  4. CD Pipeline: Deploy ARM template to Test ADF with test-specific parameter file (test storage accounts, test SQL DBs). After approval, deploy to Prod ADF with prod parameter file.
  5. Environment-specific parameters: The parameter file overrides linked service connection strings, Key Vault references, storage paths, and integration runtime names per environment.
  6. Triggers: Stop triggers before deployment, deploy, restart triggers. Use the PrePostDeploymentScript.ps1 that Microsoft provides for this.
  7. Never manually edit Prod: All changes flow through Git → Dev → Test → Prod pipeline. Any hotfix goes through a hotfix branch, not direct Prod edits.

When would you choose Hash distribution over Round-Robin or Replicated tables in a Synapse Dedicated Pool?
Distribution
▼

Synapse Dedicated Pool distributes data across 60 distributions (physical storage units). How data lands across those distributions determines query performance.

  • Hash distribution: Rows with the same hash column value always land on the same distribution. Use on large fact tables (millions+ rows) frequently joined on the same column (e.g., customer_id). When two hash-distributed tables are joined on the same column, no data movement is required — a massive performance gain.
  • Round-Robin: Rows distributed evenly across all 60 distributions sequentially. Very fast writes. Use for staging tables, temp tables, or tables without a clear join key. Bad for query performance — joins always require data movement.
  • Replicated: A full copy of the table on every compute node. Use only for small dimension tables (<2GB). Eliminates all shuffle on joins with fact tables. Avoid for large tables — 60x storage cost and slow writes on updates.
💡
Rule of thumb: large table + frequent joins → Hash on join key. Small dimension table → Replicate. Staging area → Round-Robin.
2
What is the Hierarchical Namespace (HNS) in ADLS Gen2 and how does it optimize big data operations?
Fundamentals
▼

Regular Azure Blob Storage is a flat namespace — "folders" are just prefixes in blob names. Operations like renaming or deleting a "folder" require enumerating and updating every blob with that prefix — O(n) operations at scale.

ADLS Gen2 with HNS enabled creates a true hierarchical file system like HDFS. Directories are first-class objects with their own metadata. This enables:

  • Atomic rename: Renaming a directory is a single metadata operation — O(1). Critical for Spark commit operations, which write to a temp location then rename to final.
  • Atomic delete: Delete an entire directory in one operation rather than looping through every file individually.
  • POSIX-compliant ACLs: Set read, write, and execute permissions at the directory level, not just file level.
  • Performance at scale: Dramatically faster directory-level operations when you have millions of files.
⚠️
Always enable HNS when creating storage accounts for data lake workloads. It cannot be enabled after account creation.
3
Explain Synapse Serverless SQL Pools versus Dedicated SQL Pools — when do you use each?
Architecture
▼

Serverless SQL Pool: No provisioned compute. Query files directly in ADLS (Parquet, CSV, JSON, Delta) using T-SQL with OPENROWSET or external tables. Pay per TB scanned. Great for ad-hoc exploration, data discovery, and building views for Power BI on top of the data lake. Limitation: no indexes or materialized aggregates — every query scans files.

Dedicated SQL Pool: You provision DWUs — dedicated compute and columnar storage (CCI). Supports indexes, statistics, materialized views, and workload management. Pay per DWU-hour. Best for repeated, complex BI queries with multi-user concurrency and SLA requirements.

📌
Common pattern: Serverless SQL for the Silver lake layer (ad-hoc, flexible exploration) | Dedicated Pool for the Gold layer (performance-critical BI dashboards).

What is the difference between PolyBase and COPY INTO in Synapse?
Data Loading
▼

Both load data from ADLS into a Dedicated Pool table, but they differ significantly:

PolyBase is the older approach. Requires creating an External Data Source, External File Format, and External Table as separate objects before loading. More verbose setup, supports fewer authentication methods. Still functional but is the legacy path.

COPY INTO is the modern, preferred approach. A single T-SQL statement handles everything. Supports more file formats (CSV, Parquet, ORC), more authentication methods (SAS, Managed Identity, Service Principal), and handles bad rows gracefully with an ERRORFILE option.

COPY INTO dbo.orders
FROM 'https://mystorageacct.dfs.core.windows.net/container/orders/*.parquet'
WITH (FILE_TYPE = 'PARQUET', CREDENTIAL = (IDENTITY = 'Managed Identity'));

What is workload management in Synapse and why is it important?
Performance
▼

Without workload management, a single heavy query can starve all other users of DWU resources. Synapse workload management has three components:

  • Workload groups: Define resource pools with min/max DWU percentages — e.g., ETL jobs get 50%, dashboards get 30%, ad-hoc gets 20%.
  • Workload classifiers: Route specific users, labels, or query patterns to a workload group — e.g., etl_user → ETL group; label 'dashboard' → Dashboard group.
  • Workload importance: HIGH/MEDIUM/LOW priority within the queue. High-importance queries run first when resources are contested.

How do you secure data in Azure Synapse Analytics?
Security
▼
  • Network: Deploy Synapse in a managed VNet. Use Private Endpoints to block public internet exposure. Apply Synapse Firewall rules to whitelist trusted IP ranges.
  • Authentication: Azure AD integration with AAD users and groups — avoid SQL logins where possible. Use Service Principals for automated pipelines.
  • Authorization: RBAC at the workspace level (Owner, Contributor, Reader). SQL-level roles. Column-Level Security and Row-Level Security for fine-grained access.
  • Dynamic Data Masking (DDM): Mask sensitive columns (SSN, credit card) for unauthorized users at query time. Data is not changed on disk — masking is applied at query time only.
  • Encryption: Data at rest encrypted with Azure-managed keys or Customer-Managed Keys (CMK) in Key Vault. All data in transit uses TLS.
  • Auditing: Enable Synapse auditing to write audit logs to a Storage Account or Log Analytics for compliance and forensic investigation.

1
A Databricks job is failing with Out-of-Memory (OOM) errors. How would you troubleshoot it?
Troubleshooting
▼
  1. Identify where OOM occurs: Driver OOM → you're collecting too much data to the driver (collect(), toPandas(), large broadcast). Executor OOM → individual tasks are too memory-heavy.
  2. Check Spark UI: Go to Stages → look for tasks with very large shuffle read/write or skewed task sizes.
  3. Fix data skew: Add salting for skewed joins, or enable AQE (spark.sql.adaptive.enabled=true).
  4. Increase shuffle partitions: Default is 200. For large data, increase to 400–1000 to reduce per-partition size.
  5. Upgrade cluster node type to increase executor memory, or increase spark.executor.memory in cluster config.
  6. Reduce data being processed: Push filters earlier. Use select() to drop unused columns before joins.
  7. Replace Python UDFs with native Spark functions — UDFs serialize data to a Python process, consuming extra memory.
  8. Avoid collect() on large DataFrames. Sample or aggregate first before pulling data to the driver.

How do you monitor and troubleshoot failed jobs in Azure Databricks?

What is the difference between schema inference and predefined schema in PySpark? Which is better in production?
Schema
▼

Schema inference means Spark reads a sample of the data and auto-detects column names and types. Convenient in development, but problematic in production:

  • Expensive: requires an extra scan of the data before processing begins.
  • Unreliable: inferSchema on CSV might read "123" as integer, but rows later with "123abc" fail or produce wrong types.
  • Non-deterministic: schema can silently change if source data changes, breaking downstream code without any obvious error.

Predefined schema (StructType) — you explicitly declare every column name and type before reading:

schema = StructType([
    StructField("order_id", IntegerType(), False),
    StructField("amount", DoubleType(), True),
    StructField("created_at", TimestampType(), True)
])
df = spark.read.schema(schema).json("path/to/data")
✅
Always use predefined schema in production. It's faster, predictable, and forces you to explicitly handle schema changes — which is exactly what you want in a controlled pipeline.

How would you build a reusable Python ETL framework in Databricks?
Architecture
▼
  • Config-driven: Store pipeline configs in JSON, YAML, or a control SQL table. Code reads config at runtime — add a new pipeline by adding a config row, not writing new code.
  • Separate concerns: Organize into readers/, transformers/, writers/, and utils/ (logging, audit, schema validation).
  • Base classes: Abstract base class for Reader, Transformer, Writer with standard read(), transform(df), write(df) interfaces.
  • Audit logging: Every pipeline run writes to an audit table — pipeline name, run ID, start/end time, rows read/written, status, error message.
  • Package as a wheel: Build as a Python package, publish to Databricks repos, install on clusters with %pip install.
  • Unit tests: Use pytest + chispa (PySpark DataFrame equality testing). Run in CI/CD pipeline (GitHub Actions / Azure DevOps).

How do you handle Schema Drift using ADF and Databricks?
Schema Management
▼

Schema drift = source adds/removes/renames columns unexpectedly, breaking your pipeline.

In ADF Data Flow: Enable "Allow schema drift" on the source transformation. ADF will pass through unknown columns. Use "Auto mapping" in the sink.

In Databricks:

  • Read with mergeSchema = True for Parquet — Spark merges schemas across files with different columns.
  • For Delta: use mergeSchema = true on write. Delta handles additive changes automatically; destructive changes (column removal) need explicit ALTER TABLE DROP COLUMN.
  • Add a validation step that compares the incoming schema to the expected schema and sends an alert before writing rather than silently dropping new columns.
📋
For Power BI downstream: use a view layer in Synapse or Databricks SQL that selects only the stable columns BI depends on — insulating reports from upstream schema changes.

How do you optimize a slow Databricks job? End-to-end approach.
PerformanceTuning
▼

Work through these layers in order — fix the cheapest issue first:

  1. Check Spark UI first: Go to Stages → find the bottleneck stage. Look at: task duration variance (skew?), shuffle read/write size (too much data?), spill to disk (OOM pressure?), GC time (memory tuning needed?).
  2. Fix data skew: If one task takes 10x longer than others, you have skew. Enable AQE (spark.sql.adaptive.enabled=true). For severe skew, use salting on the join key.
  3. Reduce shuffle: Increase shuffle partitions (spark.sql.shuffle.partitions) so each partition is smaller. Use broadcast join for small tables (<200MB). Push filters before joins to reduce data volume.
  4. Fix the small files problem: If reading thousands of tiny files, use OPTIMIZE (Delta) or repartition() before writing. Read files are also a performance issue — merge them first.
  5. Upgrade cluster or right-size: If spill is high → more memory per node. If task parallelism is low → more cores. Profile first, don't just throw bigger hardware at it.
  6. Use Delta Lake: If reading Parquet, switch to Delta — data skipping, Z-ordering, and statistics dramatically reduce scan size.
  7. Cache strategically: Cache DataFrames used in multiple actions. Unpersist when done.
  8. Check cluster config: Is autoscaling enabled? Is the cluster too small for the data volume on that day? Check job run history for cluster size vs data volume correlation.

How do you control Azure costs across your data platform?
Cost Optimization
▼
  • Databricks: Use spot instances for batch (90% cheaper). Use job clusters (auto-terminate) not all-purpose clusters (idle billing). Enable auto-termination (15–30 min idle). Use cluster pools for faster spin-up without idle cost. Right-size — profile job metrics and pick smallest fitting cluster.
  • Synapse Dedicated Pool: Pause when not in use (ADF pipeline or Azure Automation). Scale down DWU during off-peak. Use Serverless SQL Pool for ad-hoc exploration — pay per TB scanned, not per hour.

  • ADF: Minimize Data Flow usage (Spark cost). Prefer Copy Activity for simple ingestion. Use IR wisely — don't run SHIR on over-provisioned VMs.
  • Monitoring: Set up Azure Cost Management budgets and alerts. Tag all resources with project and environment tags. Review Azure Advisor recommendations monthly.
  • Reserved Instances: For always-on resources (ADF IR, certain Databricks clusters), commit to 1-year Reserved Instance — up to 40% savings.

How do you ensure data quality and monitoring in production pipelines?
Data Quality
▼

Data Quality Checks (at each layer):

  • Schema validation: Compare incoming schema to expected schema before writing. Alert on mismatch — don't silently drop columns.
  • Row count checks: Assert that output rows are within an expected range (e.g., ±10% of yesterday's count). Zero rows is almost always a bug.
  • Null checks: Assert that critical columns (primary keys, join keys, foreign keys) have zero nulls.
  • Uniqueness checks: Assert no duplicate primary keys in the target table after load.
  • Referential integrity: Assert that all foreign keys have matching records in dimension tables (no orphaned fact records).
  • Business rule validation: Amount >= 0, dates are in valid range, status is in allowed set of values.

Tools in Azure:

  • Great Expectations or Deequ (Databricks): Programmatic data quality rule engines that generate quality reports.
  • Azure Monitor + Log Analytics: Ship custom metrics (row counts, null counts, error counts) from pipelines. Set alert rules — trigger email/Teams/PagerDuty when a metric breaches threshold.
  • ADF Monitor: Pipeline run history, activity-level error messages, built-in alerts on pipeline failure.
  • Audit table: Write pipeline run metadata (pipeline name, run ID, start/end, rows in/out, status, error) to a central audit SQL table. Build a Power BI dashboard on top for visibility.

Describe a production issue you handled end-to-end.
Scenario
▼
💡
This is a behavioral question. Use the STAR format: Situation → Task → Action → Result. Structure your answer around a real incident — interviewers value specific details over generic answers. Below is a model answer framework.

Example structure to adapt:

  • Situation: "Our nightly ADF pipeline that loaded 15M sales records from on-prem SQL to Synapse started failing intermittently at 2 AM — only on certain days."
  • Task: "I was on-call. I had to diagnose, fix, and prevent recurrence before the business opened at 9 AM."
  • Action: (1) Checked ADF Monitor — error was a connection timeout on the SHIR. (2) SSH'd to the SHIR machine — disk was 98% full due to accumulated ADF log files. (3) Cleared logs, freed disk. (4) Pipeline re-ran successfully. (5) Root cause: no log rotation policy on SHIR. How do you handle Data Skewness in a PySpark cluster? Explain Salting.
Performance
▼

Data skew occurs when data is not evenly distributed across partitions. One partition may have 10 million rows while others have 10,000 — that one task takes 100x longer and becomes the bottleneck of the entire stage.

  • Salting (manual): Append a random integer (0 to N-1) to the join key on the skewed side. Explode the small side N times with each salt value. Join on the salted composite key. Drop the salt column after joining. This distributes one hot key across N partitions.
  • AQE: Enable spark.sql.adaptive.enabled=true and spark.sql.adaptive.skewJoin.enabled=true. Spark 3.x automatically detects and splits skewed partitions at runtime — no code changes needed.
  • Broadcast join: If the smaller side is <~200MB, broadcast it to eliminate the shuffle entirely.
  • Repartition on a composite key that distributes more evenly before the join.

Explain the difference between Vectorized UDF (pandas_udf) and standard UDF. Which is more efficient?
Performance
▼

Standard UDF: A Python function registered to run row-by-row. Spark cannot optimize it — it's a black box. Each row is serialized from JVM → Python process → processed → serialized back. This overhead is enormous, and UDFs disable Catalyst optimizations like predicate pushdown.

Vectorized UDF (pandas_udf): Data is batched into Pandas Series using Apache Arrow for zero-copy serialization. Much faster — typically 3-10x compared to row-at-a-time UDFs.

from pyspark.sql.functions import pandas_udf
import pandas as pd

@pandas_udf("double")
def multiply_by_two(s: pd.Series) -> pd.Series:
    return s * 2

🏆

Explain partitioning and bucketing in Apache Spark and when to use each.
Storage
▼

Partitioning splits data into directories based on column values — e.g., partitionBy("year", "month") creates year=2024/month=01/ folders. Queries filtering on the partition column skip entire directories (partition pruning). Best for low-cardinality columns like date, region, or status. High-cardinality columns (user_id) → millions of tiny files (small files problem).

Bucketing hashes data into a fixed number of files (buckets) based on a column. Files are in one directory; the structure is internal metadata. Key benefit: when two tables are bucketed on the same column with the same bucket count, Spark can join them without any shuffle.

df.write.bucketBy(64, "customer_id").sortBy("customer_id")\
    .saveAsTable("orders_bucketed")

Partitioning is for read-time pruning (filter performance). Bucketing is for join performance (shuffle elimination). They can be combined.

What is the difference between cache() and persist()? When should you use DISK_ONLY?
Memory
▼

cache() is shorthand for persist(StorageLevel.MEMORY_AND_DISK). It tries to store the DataFrame in memory and spills to disk if memory is insufficient.

persist() is the full API — choose the storage level explicitly:

  • MEMORY_ONLY: Fastest. If it doesn't fit, partitions are dropped and recomputed on access — no disk fallback.
  • MEMORY_AND_DISK: Default of cache(). Spills excess partitions to disk instead of recomputing. Safe for most use cases.
  • DISK_ONLY: Never uses memory. Use when the DataFrame is too large to fit in memory AND you want to avoid OOM errors. Slower but prevents crashes for extremely large intermediate results.
  • MEMORY_AND_DISK_SER: Serializes objects (smaller footprint) at cost of CPU for ser/deser.
⚠️
Always call unpersist() when done to free resources. Never cache a DataFrame used only once — it's pointless overhead.

What is Adaptive Query Execution (AQE)? How does it help at runtime?
Optimization
▼

Traditional Spark optimization happens at query parse time based on statistics that may be stale or missing. AQE (Spark 3.0+) re-optimizes the query plan at runtime using actual shuffle statistics.

  • Dynamic coalescing of shuffle partitions: If data is small, AQE merges small partitions → fewer tasks, less overhead.
  • Dynamic switch to broadcast join: If at runtime one side turns out small enough to broadcast, AQE switches from sort-merge to broadcast hash join mid-query.
  • Dynamic skew join handling: AQE detects skewed partitions (5x larger than median) and splits them into parallel sub-tasks.

Enable: spark.conf.set("spark.sql.adaptive.enabled", "true"). In Databricks, AQE is enabled by default from DBR 7.3+.

What is Spark Spill? How do you identify it in the Spark UI and resolve it?

▼

Identify: Spark UI → Stages tab → look for non-zero Shuffle Spill (Memory) and Shuffle Spill (Disk) columns. Also check Stage detail → Task metrics for high "Disk bytes spilled".

Explain Spark Architecture — Driver, Executors, Cluster Manager, DAG
Architecture
▼

Core components:

  • Driver: The JVM process that runs your Spark application's main(). It creates the SparkContext, builds the DAG (Directed Acyclic Graph) of transformations, divides the DAG into stages and tasks, and schedules tasks on executors. The driver has a finite memory — never collect() large data to it.
  • Executor: Worker JVM processes that run tasks. Each executor has a fixed number of CPU cores (tasks run in parallel = cores) and memory (execution + storage). Executors live for the lifetime of the application.
  • Cluster Manager: Allocates resources (executors) to the Spark application. In Databricks: YARN (legacy) or Kubernetes. In standalone: Spark's built-in manager. The driver requests resources from the cluster manager.
  • DAG (Directed Acyclic Graph): Spark builds a logical plan of transformations lazily. When an action (count(), write(), collect()) is triggered, the Catalyst optimizer converts the logical plan into a physical plan, splits it into stages at shuffle boundaries, and each stage into tasks (one task per partition).
  • RDD → DataFrame → Dataset: RDDs are the low-level distributed data structure. DataFrames are RDDs with schema — the Catalyst optimizer can reason about them. Always use DataFrames/SQL in production, not raw RDDs.
  • Lazy evaluation: Transformations (filter(), join(), select()) don't execute immediately. They build the DAG. Only actions trigger actual computation. This allows Catalyst to optimize the full plan before executing anything.

Catalyst Optimizer & Tungsten Engine — what do they do?
Internals
▼

Catalyst Optimizer is Spark SQL's query optimizer. It works in 4 phases:

  1. Analysis: Resolves column names, table references, and data types against the catalog. Catches "column not found" errors here.
  2. Logical Optimization: Applies rule-based rewrites — predicate pushdown (move filters as early as possible), projection pruning (drop unused columns early), constant folding (1+1 → 2), subquery elimination.
  3. Physical Planning: Converts logical plan into physical plan — chooses join strategy (broadcast vs sort-merge vs hash), decides partition count, selects sort algorithms.
  4. Code Generation: Generates optimized JVM bytecode for each physical operator — eliminates virtual function calls, enables CPU-level optimizations.

Tungsten Engine is Spark's memory and CPU optimization layer, operating below Catalyst:

  • Off-heap memory management: Bypasses JVM garbage collection by managing memory directly. Reduces GC pauses dramatically.
  • Cache-aware computation: Designs algorithms (sort, hash) to maximize CPU L1/L2 cache usage — orders of magnitude faster than cache-miss-heavy algorithms.
  • Whole-stage code generation: Instead of one function per operator (many virtual calls), generates a single optimized loop for the entire pipeline. Dramatically reduces CPU overhead.
  • Binary row format: Stores data in compact binary format in memory — smaller footprint, faster serialization than Java objects.
💡
Catalyst makes your SQL/DataFrame code correct and logically optimal. Tungsten makes the physical execution fast at the CPU/memory level. Together they make Spark DataFrames much faster than raw RDDs.

Spark Join Strategies — Broadcast, Sort-Merge, Shuffle Hash, Nested Loop
Join Strategies
▼
  • Broadcast Hash Join (BHJ): Small table broadcast to all executors. Large table partitions join locally — no shuffle. Fastest join. Use when one side < spark.sql.autoBroadcastJoinThreshold (default 10MB). Force with broadcast(df) hint.
  • Sort-Merge Join (SMJ): Both sides shuffled by join key, then sorted, then merged. Default for large-large joins. Both sides must be sortable. Reliable but expensive (full shuffle of both sides).
  • Shuffle Hash Join (SHJ): Both sides shuffled by join key. Smaller side hashed into a hash table. Larger side probes it. Faster than SMJ if one side fits in executor memory, but can OOM if the hash table is too large. Spark uses this when sortability is not available.
  • Broadcast Nested Loop Join (BNLJ): Used for non-equi joins (joins without = condition — e.g., range joins, cross joins with filters). One side broadcast, the other loops through it for every row. Very slow for large tables — avoid if possible. Spark falls back to this when no other strategy works.

How Spark chooses: BHJ (if small enough) → SMJ (if sortable) → SHJ (if not sortable but fits in memory) → BNLJ (last resort). AQE can switch from SMJ to BHJ at runtime if statistics show one side is smaller than expected.

⚠️
If you see "BroadcastNestedLoopJoin" in your explain plan, investigate immediately — it often means a missing join condition or an accidental cross join that will kill your job on large data.

Distributed Variables in Spark — Broadcast Variables & Accumulators
Spark Internals
▼

Broadcast Variables: Normally, any variable used inside a transformation is serialized and sent to each task (once per task, potentially thousands of copies). Broadcast Variables send the data once to each executor (not each task) and cache it in memory there. All tasks on that executor share one copy.

lookup_dict = {"NY": "New York", "CA": "California"}
broadcast_dict = spark.sparkContext.broadcast(lookup_dict)

# Inside transformation - all tasks on same executor share one copy
df.rdd.map(lambda row: broadcast_dict.value.get(row["state"], "Unknown")).toDF()

Use broadcast variables for large lookup dictionaries, config maps, or ML models that need to be available on every executor. For DataFrames, use the broadcast() function hint instead.

Accumulators: Variables that tasks can only add to (write), and only the driver can read. Used to track counters across distributed tasks — e.g., count bad records, count rows processed per pipeline stage.

bad_records = spark.sparkContext.accumulator(0)

def process_row(row):
    if row["amount"] is None:
        bad_records.add(1)
        return None
    return row

df.foreach(process_row)
print(f"Bad records found: {bad_records.value}")
⚠️
Accumulators can be double-counted if a task is retried (Spark re-runs failed tasks). Use them for approximate monitoring metrics, not for business-critical counting. For exact counts, use DataFrame aggregations.

How do you implement parallel processing in Python for ETL workloads? How do you structure a reusable Python ETL framework?

# Process 4 tables in parallel (I/O bound → ThreadPool)
with ThreadPoolExecutor(max_workers=4) as executor:
    futures = {executor.submit(process_table, t): t for t in tables}

        print(future.result())

Partitioning strategies and how they impact performance
Partitioning
▼

Physical partitioning (on disk): Splitting data into directories based on column values — partitionBy("year","month"). Queries filtering on those columns skip irrelevant directories (partition pruning). Performance is excellent for the partition columns; no benefit for other columns.

Choosing partition columns: Low cardinality (date, region, status) — good. High cardinality (user_id, order_id) — bad (millions of tiny directories = small files problem, driver overwhelmed by metadata).

In-memory partitioning (Spark): Controls how data is split across executor tasks during processing.

  • spark.sql.shuffle.partitions: Number of partitions after a shuffle (default 200). Too few → large partitions → spill/OOM. Too many → tiny partitions → task overhead. Rule of thumb: target 100-200MB per partition.
  • repartition(n, col): Full shuffle to redistribute evenly, optionally on a column for co-location before a join.
  • coalesce(n): Reduce partition count without shuffle — cheap but potentially uneven.

Synapse/SQL Server table distribution: Hash (join performance), Round-Robin (staging/loading), Replicated (small dimension tables). This is a physical partitioning decision that affects every query permanently — choose carefully at table creation.

💡
Good partitioning = the right columns pruned at query time + balanced partition sizes in memory. Bad partitioning = either too few (skew/OOM) or too many (small files/overhead).

What is your approach to designing a fault-tolerant, scalable ETL/ELT pipeline in Azure? What tools and tradeoffs?
Architecture
▼

Core design principles:

  • Idempotency: Every pipeline run must be safe to re-run. Use watermarks, MERGE, or partition overwrite — never append-only without dedup.

  • ADF (orchestration): Managed, UI-driven, easy CI/CD. Tradeoff: limited complex logic — move that to Databricks notebooks.

  • Delta Lake (storage format): ACID, time travel, schema enforcement. Tradeoff: overhead vs raw Parquet for very simple pipelines.
  • Synapse (serving): Excellent for BI workloads and multi-user concurrency. Tradeoff: DWU cost is always-on; Serverless SQL is cheaper for ad-hoc but slower for repeated queries.

How do you implement Row-Level Deletions in Delta Lake without rewriting the entire partition?
Delta Operations
▼

Traditional Parquet is immutable — deleting a row requires rewriting the entire file. Delta Lake introduces row-level deletes via a two-step mechanism:

from delta.tables import DeltaTable

dt = DeltaTable.forPath(spark, "/mnt/delta/orders")

Explain the Data Skipping mechanism in Delta Lake. How does it differ from traditional SQL indexing?
Optimization
▼

When you read from a Delta table with a filter (e.g., WHERE event_date = '2024-01-15'), Delta reads the transaction log's file-level statistics before scanning any data. For each Parquet file, the log stores: min value, max value, and null count per column.

If a file's event_date range is 2024-01-10 to 2024-01-12, Delta knows it cannot possibly contain 2024-01-15 and skips it entirely — no I/O, no task created.

vs. traditional SQL index: A B-tree index points to exact row locations — ideal for OLTP single-row lookups. Delta's data skipping is coarser (file-level, not row-level) but is designed for analytical scans across billions of rows in object storage where random I/O is impractical.

💡
Z-Ordering colocates related data within files: OPTIMIZE table ZORDER BY (customer_id, event_date). This makes min/max statistics tight and effective — maximizing data skipping hits on those columns.

How do you choose between Partitioning and Z-Ordering for a 10TB dataset?

▼

These solve different problems and work best together:

Partitioning creates physical directory splits. A query with WHERE year=2024 reads only that folder — extremely effective for the partition column. Problem: high-cardinality columns (user_id with millions of values) → millions of tiny directories.

Z-Ordering works within partitions. It reorders rows so rows sharing values in Z-ordered columns are stored together in the same files, making data skipping (min/max) effective for those columns.

  • Pick 1-2 low-cardinality columns that appear in almost every query filter → Partition by these (e.g., date, region).
  • Pick 2-3 higher-cardinality columns that appear in many filters but aren't suitable for partitioning → Z-Order by these (e.g., customer_id, product_id).
  • Never partition by a high-cardinality column — this creates the severe small files problem.

How would you optimize a slow SQL query? What steps and tools do you use?
Performance
▼
  1. Read the execution plan first: EXPLAIN ANALYZE SELECT ... — look for full table scans, sort operations, nested loops, and large gaps between estimated and actual row counts.
  2. Check indexes: Is there an index on the filter columns (WHERE, JOIN ON)? Are existing indexes being used or bypassed by the optimizer?
  3. Reduce the dataset early: Push WHERE filters as early as possible. Avoid SELECT * — select only the columns needed.
  4. Avoid functions on indexed columns: WHERE YEAR(created_at) = 2024 prevents index use. Rewrite as: WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'

  7. Avoid correlated subqueries: Replace them with JOINs or CTEs — correlated subqueries execute once per row in the outer query.
  8. In Synapse: Check distribution skew with DBCC PDW_SHOWSPACEUSED, update statistics, consider BROADCAST hints on small dimension tables.

Explain database indexing and its impact on performance.
Fundamentals
▼

An index is a separate data structure (usually a B-tree) that stores a sorted copy of one or more columns along with pointers to the original rows. Instead of scanning every row, the database finds matching rows in O(log n) time.

  • Clustered index: The table's physical row order matches the index. One per table. Best for range queries on the clustered key.
  • Non-clustered index: A separate structure pointing into the table. Multiple allowed. Best for frequently-filtered non-PK columns.
  • Covering index: Includes all columns needed by a query (using INCLUDE). The query is satisfied from the index alone — no table lookup required.
  • Columnstore index: Stores data column by column, highly compressed. Ideal for analytical aggregations over millions of rows.
⚖️
Indexes speed up reads but slow down writes — the index must be maintained on every INSERT, UPDATE, and DELETE. Index only columns that appear in WHERE, JOIN ON, and ORDER BY clauses of slow queries.

Explain the MERGE statement. How is it different from UPSERT?
Code
▼

UPSERT is a concept: insert if a record doesn't exist, update if it does. MERGE is the SQL statement that implements this — and more. MERGE is a superset of UPSERT — it can handle UPDATE, INSERT, and DELETE in one atomic statement.

MERGE INTO target_table AS t
USING source_table AS s
ON t.order_id = s.order_id
WHEN MATCHED AND s.amount != t.amount THEN
    UPDATE SET t.amount = s.amount, t.updated_at = GETDATE()
WHEN NOT MATCHED BY TARGET THEN
    INSERT (order_id, amount, created_at)
    VALUES (s.order_id, s.amount, GETDATE())
WHEN NOT MATCHED BY SOURCE THEN
    DELETE;  -- rows in target not in source → delete

The WHEN NOT MATCHED BY SOURCE THEN DELETE clause goes beyond UPSERT — it's a full sync that removes rows from target that no longer exist in source. MERGE is atomic — either all three operations succeed or none do.

Indexes in SQL — types, internals, when to use / avoid
IndexesCore Concept
▼

An index is a separate data structure (B-tree or other) that stores a sorted copy of one or more columns with pointers to original rows. It converts O(n) table scans into O(log n) lookups.

Types:

  • Clustered Index: Determines the physical order of rows in the table. One per table. Rows are stored in the B-tree leaf nodes. Range queries on the clustered key are extremely fast. SQL Server PKs are clustered by default.
  • Non-Clustered Index: Separate structure with key + pointer (row locator) back to the table. Multiple allowed. Best for columns frequently in WHERE, JOIN ON, ORDER BY that aren't the PK.
  • Covering Index (INCLUDE): A non-clustered index that includes extra columns (via INCLUDE clause) beyond the key. Query can be satisfied entirely from the index — no lookup to the main table ("index-covered query"). Fastest reads for specific query patterns.
  • Composite Index: Index on multiple columns. Column order matters — the index can only be used for queries that filter on a left-prefix of the index columns. INDEX(a, b, c) can serve WHERE a=?, WHERE a=? AND b=?, but NOT WHERE b=?.
  • Columnstore Index: Stores data column-by-column (vertically), highly compressed. Optimal for analytics — aggregating millions of rows on a few columns. Used by default in Synapse Dedicated Pool (CCI = Clustered Columnstore Index). Not suited for row-level lookups.
  • Filtered Index: Index with a WHERE clause — only indexes rows matching the filter. Smaller, faster for specific query patterns (e.g., index only status = 'ACTIVE' rows).

When NOT to index: Columns with very low cardinality (e.g., gender — only 2 values, a scan is faster than an index on 50% of a table). Tables with very frequent writes — every INSERT/UPDATE/DELETE maintains all indexes. Staging/temp tables — not worth the overhead.

⚠️
Common pitfalls: functions on indexed columns prevent index use (WHERE YEAR(order_date)=2024 → full scan). Leading wildcards in LIKE prevent index use (LIKE '%keyword' → full scan, but LIKE 'keyword%' uses the index).

▼

Key insight: for consecutive dates, login_date - ROW_NUMBER() gives the same constant value for all days in a streak. That constant is the "streak group key".

🗄 SQL
WITH deduped AS (
  -- Remove duplicate logins on same day
  SELECT DISTINCT employee_id,
    CAST(login_date AS DATE) AS login_date
  FROM employee_logins
),
numbered AS (
  SELECT employee_id, login_date,
    ROW_NUMBER() OVER (PARTITION BY employee_id ORDER BY login_date) AS rn
  FROM deduped
),
grouped AS (
  SELECT employee_id, login_date,
    -- For consecutive days, this value is constant
    DATEADD(DAY, -rn, login_date) AS streak_grp
  FROM numbered
)
SELECT
  employee_id,
  streak_grp,
  MIN(login_date) AS streak_start,
  MAX(login_date) AS streak_end,
  COUNT(*)        AS streak_length
FROM grouped
GROUP BY employee_id, streak_grp
HAVING COUNT(*) >= 2          -- only streaks of 2+ days
ORDER BY employee_id, streak_length DESC;

-- To find the LONGEST streak per employee:
-- Wrap above in another CTE and use MAX(streak_length)

from pyspark.sql.functions import (
  col, row_number, date_add, min as _min,
  max as _max, count, to_date, datediff, lit
)
from pyspark.sql.window import Window

How do you use Azure Key Vault and Managed Identities to eliminate hardcoded credentials?
Security
▼

Azure Key Vault stores all secrets (passwords, connection strings, API keys) securely. Access is controlled via RBAC.

Managed Identity (MI) is an Azure AD identity automatically assigned to an Azure resource (ADF, Databricks, App Service). The identity has no password — Azure manages the credentials internally. You grant this identity access to Key Vault, and the resource authenticates using its MI, fetches the secret, and uses it — all without any hardcoded credentials.

  • In ADF: ADF has a System-Assigned MI automatically. Grant it the "Key Vault Secrets User" role. In Linked Service, reference: @Microsoft.KeyVault(VaultName=myVault;SecretName=sqlPassword). ADF fetches the secret at runtime — no password in ADF config.

  • For ADLS access: Configure Spark session to use OAuth with MsiTokenProvider. Grant the Databricks cluster's MI the "Storage Blob Data Contributor" role on ADLS. No storage keys needed anywhere.

How do you manage secrets securely in a production pipeline?
Best Practice
▼
  • Never hardcode secrets in notebooks, config files, or pipeline definitions — even temporarily. They end up in version control and are extremely difficult to fully remove from git history.
  • Key Vault for all secrets. Rotate secrets in Key Vault without touching your code — services fetch the latest value on the next run automatically.
  • Prefer Managed Identity over Service Principals over passwords. MIs have no credentials to leak. SPs have client secrets that expire and must be rotated. SQL passwords are the least secure option.
  • Databricks secret scopes: dbutils.secrets.get() — Databricks automatically redacts these values from notebook output and cluster logs.
  • Environment-specific secrets: Separate Key Vaults (or secret versions) per environment — dev, staging, prod. Pipelines reference the environment-appropriate vault via a parameter.
  • Audit Key Vault access: Enable diagnostic logs → Log Analytics. Alert on unexpected access patterns or access from unexpected principals.
  • Least privilege: Grant each service identity only the specific secrets it needs — not blanket "Get All Secrets" on the entire vault.

How do you implement Column-Level Security and Row-Level Security in a Lakehouse?
Security
▼

Column-Level Security (CLS): Prevent specific users or roles from reading specific columns entirely.

-- Synapse Dedicated Pool
DENY SELECT ON dbo.customers(ssn, credit_card) TO analyst_role;
GRANT SELECT ON dbo.customers(customer_id, name, email) TO analyst_role;

-- Databricks Unity Catalog
GRANT SELECT ON TABLE customers TO analyst_role;

Row-Level Security (RLS): Restrict which rows a user can see based on their identity. In Synapse, create a security predicate function that evaluates the current user's access context, then bind it to the table as a SECURITY POLICY with FILTER PREDICATE — every query on the table automatically runs through the predicate.

In Databricks Unity Catalog: Define a column masking function and attach it to a column. Any query on that column runs through the masking function based on the caller's group membership — fine-grained, centrally managed access control across all tools querying through Unity Catalog.

How do you handle Schema Evolution without breaking downstream Power BI reports?
Architecture
▼

Core principle: insulate consumers from upstream schema changes using a stable interface layer.

  • Views as a contract layer: Power BI connects to a view (Synapse serverless view or Databricks SQL view), not directly to the Delta table. When upstream schema changes, update the view to maintain the same output columns — Power BI sees no change.
  • Semantic model isolation: In Power BI, use a semantic model (dataset) as the interface. Reports connect to the model, not the underlying source directly. Update the model's queries when schema changes, without touching any reports.
  • Additive changes are safe: Adding new columns to a Delta table doesn't break existing queries. Treat column removal or renaming as breaking changes requiring a deprecation period.
  • Column aliasing: If a source column is renamed, add an alias in the view to maintain backward compatibility during the transition period.
  • Schema change alerts: Add a validation step in your pipeline that compares the incoming schema to the expected schema and sends an alert before writing — giving you time to update the interface layer before new schema lands in production.

What are window functions in SQL and when would you use them?
ConceptWindow Functions
▼

A window function performs a calculation across a set of rows related to the current row — without collapsing them into a single output row like GROUP BY does. The "window" is defined by OVER (PARTITION BY ... ORDER BY ...).

Use window functions when you need both the individual row AND an aggregation — e.g., "show each employee's salary alongside their department average."

  • Ranking: ROW_NUMBER(), RANK(), DENSE_RANK() — rank rows within a group.
  • Running totals / moving averages: SUM() OVER, AVG() OVER with ORDER BY.
  • Lead/Lag: LAG(col, 1) — previous row's value; LEAD(col, 1) — next row's value. Great for month-over-month comparisons.
  • NTILE(n): Divide rows into n equal buckets (e.g., quartiles, deciles).
  • FIRST_VALUE / LAST_VALUE: First or last value in the window frame.
🗄 SQL — Window Function Examples
SELECT employee_id, department, salary,
  -- Rank within department
  RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank,
  -- Running total per department
  SUM(salary) OVER (PARTITION BY department ORDER BY employee_id
                  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_total,
  -- Department average (no GROUP BY needed)
  AVG(salary) OVER (PARTITION BY department) AS dept_avg,
  -- Previous employee's salary
  LAG(salary, 1) OVER (PARTITION BY department ORDER BY salary DESC) AS prev_salary
FROM employees;

Explain normalization vs denormalization — when would you use each in a data engineering context?
DB Design
▼

Normalization eliminates redundancy by splitting data into multiple related tables (1NF → 2NF → 3NF). Each piece of information is stored exactly once. Benefits: consistent writes, smaller storage, no update anomalies. Best for OLTP systems (transactional databases — banking, e-commerce orders).

Denormalization intentionally introduces redundancy by merging tables or pre-computing aggregations. Benefits: fewer JOINs, faster reads, simpler queries. Best for OLAP systems (data warehouses, reporting) where read performance matters more than write efficiency.

  • Data Engineering rule of thumb: Normalize at the source/Bronze layer (preserve raw fidelity). Denormalize at the Gold layer (optimize for BI queries — flat wide tables, pre-joined dimensions).
  • Synapse Dedicated Pool: Replicated dimension tables (fully denormalized, no JOINs needed) + Hash-distributed fact tables = classic denormalized star schema.
  • Delta Lake: Use Delta tables at Silver (normalized, ACID) → Gold (wide denormalized tables optimized for reporting).
📌
In most modern data lakehouses: Normalize for correctness (Bronze/Silver) → Denormalize for performance (Gold/Serving layer).

A SQL query is running very slow on a large table. How would you troubleshoot and optimize it? You receive duplicate data from a source system — how do you handle it in your pipeline?
OptimizationPipeline Design
▼

Slow query — troubleshooting order:

  1. Run EXPLAIN / EXPLAIN ANALYZE — find the most expensive node (full table scan, sort, nested loop join).
  2. Check for missing indexes on filter and join columns.
  3. Check statistics are fresh — stale stats lead to bad execution plans. Run UPDATE STATISTICS or ANALYZE.
  4. Rewrite predicate — remove functions on indexed columns, use range conditions instead of BETWEEN on dates.
  5. Check JOIN order — filter the smallest set first before joining large tables.
  6. Add covering index if the query selects few columns from a heavily filtered table.
  7. Partition pruning — ensure the query's WHERE clause aligns with partition keys in Synapse or Delta.
  8. In Synapse: check for data skew with DBCC PDW_SHOWSPACEUSED, update statistics, add BROADCAST hints on small tables.

Handling duplicate data from source in a pipeline:

  • At ingestion (Bronze): Load raw data as-is — never deduplicate at Bronze. Log the duplicate count as an audit metric.
  • At Silver (cleansing): Deduplicate using ROW_NUMBER() or dropDuplicates() on natural keys. Define a clear "winner" rule (latest timestamp, highest sequence number).
  • Use Delta MERGE into Silver/Gold with WHEN MATCHED THEN UPDATE — naturally idempotent, handles re-delivered records.
  • Add a data quality check after dedup — assert uniqueness on the key column and alert if violation detected.
  • Alert source team — duplicate data from source is a data contract violation. Raise it so the root cause is fixed upstream.

🗄 SQL — NULL Functions

SELECT COALESCE(phone, mobile, 'No Contact') AS contact
FROM customers;

-- (avoid divide-by-zero)
SELECT revenue / NULLIF(units, 0) AS revenue_per_unit
FROM sales;

-- IS NULL / IS NOT NULL (never use = NULL)
SELECT * FROM orders WHERE discount IS NULL;

-- NULL in aggregations (NULL is ignored by SUM, AVG)
-- but COUNT(*) counts NULLs, COUNT(col) does not!
SELECT
  COUNT(*)          AS total_rows,      -- includes NULLs
  COUNT(discount)   AS rows_with_disc,  -- excludes NULLs
  AVG(discount)     AS avg_disc,        -- ignores NULLs
  AVG(COALESCE(discount, 0)) AS avg_disc_with_zero
FROM orders;

-- NULL in JOIN: NULL != NULL, so NULLs never join
-- Use IS NULL check after a LEFT JOIN to find unmatched
SELECT o.*, c.name
FROM orders o
LEFT JOIN customers c ON o.customer_id = c.customer_id
WHERE c.customer_id IS NULL;  -- orders with no customer!

What is Data Modelling? Why does it matter in data engineering?
Fundamentals
▼

Data modelling is the process of defining the structure, relationships, and constraints of data in a system. It answers: what data do we store, how is it related, and how do we organize it for efficient use?

In data engineering, good modelling directly impacts: query performance (fewer joins, better pruning), storage efficiency (no redundancy), data quality (enforced constraints), and downstream clarity (analysts and BI tools can self-serve without engineering help on every query).

  • Conceptual model: High-level entities and relationships (ERD) — "Customers place Orders containing Products". Business language, no technical detail.
  • Logical model: Entities, attributes, data types, primary and foreign keys — technology agnostic.
  • Physical model: Actual tables, columns, indexes, partitions, distributions — specific to the target technology (Synapse, Delta Lake, PostgreSQL).
2
Star Schema vs Snowflake Schema vs One Big Table (OBT)
Schema Design
▼

Star Schema: One central fact table (measurements/events/transactions) surrounded by denormalized dimension tables. Each dimension table connects directly to the fact table. Simple JOIN path, fast queries, easy for BI tools to understand. Best for Synapse, Redshift, traditional DW.

Snowflake Schema: Like star schema, but dimension tables are normalized — dimensions reference sub-dimension tables (e.g., dim_product → dim_category → dim_subcategory). Less redundancy, smaller storage, but more JOINs required. Harder for BI tools to auto-generate queries.

One Big Table (OBT): All dimensions pre-joined into the fact table — one massive denormalized table. No JOINs needed for queries = maximum read speed. Best for Lakehouse Gold layer serving BI dashboards. Downside: massive storage redundancy, difficult to maintain when dimensions change.

📌
Modern lakehouse pattern: Silver = normalized Star Schema (Delta tables, ACID, flexible). Gold = OBT or Star Schema (pre-joined, optimized for Power BI / Tableau reads). Serverless SQL views bridge Silver → Gold without duplicating data.
3
Fact Tables vs Dimension Tables — types and design rules
Star Schema
▼

Fact Tables: Store measurements, events, or transactions. Rows are additive (you can SUM them). Very wide (many metrics) and very tall (billions of rows). Always contain foreign keys to dimension tables + metric columns (amount, quantity, duration).

Types of fact tables:

  • Transaction fact: One row per event (each order, each payment, each click). Most common. Grain = one transaction.
  • Periodic snapshot: One row per time period per entity (monthly account balance, weekly inventory level). Good for trend analysis.
  • Accumulating snapshot: One row per pipeline/process instance that gets updated as milestones are hit (order placed → shipped → delivered → returned). One row follows the entity's entire lifecycle.
  • Factless fact: No numeric measures — records that an event happened (student attended class, product was eligible for promotion). Useful for counting relationships.

Dimension Tables: Describe the "who, what, where, when, why" context of facts. Smaller, slowly changing. Contain surrogate key + natural key + descriptive attributes. Types: Conformed (shared across multiple fact tables), Degenerate (dimension key in fact table with no separate dim table, e.g., order number), Junk (low-cardinality flags grouped into one table).

4
Normalization (1NF, 2NF, 3NF) — explain each with an example
Normalization
▼

1NF (First Normal Form): Every column must contain atomic (indivisible) values. No repeating groups. Each row must be unique.

  • Violation: customer(id, name, phone1, phone2, phone3) — repeating groups for phone numbers.
  • Fix: customer(id, name) + customer_phone(customer_id, phone)

2NF (Second Normal Form): Must be in 1NF, AND every non-key column must depend on the WHOLE primary key (no partial dependencies). Only matters when PK is composite.

  • Violation: order_item(order_id, product_id, quantity, product_name) — product_name depends only on product_id, not on the full composite key.
  • Fix: Move product_name to a product(product_id, product_name) table.

3NF (Third Normal Form): Must be in 2NF, AND no transitive dependencies (non-key columns must not depend on other non-key columns).

  • Violation: employee(id, name, zip_code, city) — city depends on zip_code, not on id.
  • Fix: employee(id, name, zip_code) + zip_code(zip_code, city)
📌
In data engineering: normalize for OLTP (transactional systems — 3NF minimizes update anomalies). Denormalize for OLAP (analytical — fewer JOINs = faster aggregations on billions of rows).
5
Slowly Changing Dimensions (SCD Types 0, 1, 2, 3, 4, 6)
SCD
▼

SCDs handle how dimension attributes change over time. Choosing the right type is a business decision about history preservation vs complexity.

  • Type 0: Never change. Ignore updates. Used for static reference data (country codes, currency codes).
  • Type 1: Overwrite the old value. No history. Simple. Use when historical accuracy doesn't matter (correcting typos, fixing data errors).
  • Type 2 (most common): Add a new row for each change. Preserve full history. Columns: effective_from, effective_to, is_current. Fact tables join on surrogate key — they point to the exact dimension row that was current at transaction time. Historical analysis is fully accurate.
  • Type 3: Add a new column for the previous value (e.g., current_region, prev_region). Preserves only one level of history. Simple but limited — only one change back.
  • Type 4: Separate history table — current table has only current values, history table has all previous versions. Fast reads on current, full history still available separately.
  • Type 6 (Hybrid 1+2+3): SCD2 row versioning + a current_value column on all rows. Supports "what was the value then" (Type 2) AND "what is the current value" (Type 1) in a single query without self-joining.

How do you handle Many-to-Many relationships in a Star Schema without creating a Cartesian product?
Advanced ModellingScenario
▼

A many-to-many relationship (e.g., one Order can have many Promotions, one Promotion can apply to many Orders) creates a Cartesian product if you JOIN directly — every order row multiplied by every matching promotion row = duplicated facts (inflated revenue totals).

Solution: Bridge Table (associative entity)

Create a bridge table between the fact table and the many-side dimension:

  • fact_orders → bridge_order_promotions(order_id, promotion_id, weight_factor) → dim_promotions
  • The weight_factor column (e.g., 1/count_of_promotions) allows proportional allocation — when aggregating, multiply the metric by weight_factor to avoid double-counting.

Example: Order worth ₹1000 has 2 promotions applied. Bridge table has 2 rows with weight_factor = 0.5 each. When summing revenue by promotion: ₹1000 × 0.5 = ₹500 per promotion. Total = ₹1000 ✓ (not ₹2000).

Alternative — Multi-valued dimension degenerate approach: If cardinality is low (e.g., max 3 promotions per order), store as a delimited string in the fact table and use string-splitting functions in queries. Simpler but less elegant.

⚠️
Never JOIN a fact table directly to a many-side dimension without a bridge table. The fan-out will inflate all numeric measures and produce silently wrong analytics.

How do you implement Dynamic Data Masking in a Lakehouse to protect PII data?

▼

PII (Personally Identifiable Information) masking ensures sensitive data (SSN, email, phone, credit card) is visible only to authorized users, even when the underlying data is stored in plain text.

Azure SQL / Synapse — Dynamic Data Masking (DDM):

Explain the Data Skipping mechanism in Delta Lake vs traditional SQL indexing
Delta LakePerformance
▼

Delta Lake Data Skipping: For each Parquet file in a Delta table, the transaction log stores file-level statistics: minValues, maxValues, and nullCount per column (for the first 32 columns by default). When you run a query with a filter, Delta checks these stats before opening any file. If a file's min/max range doesn't contain the filter value, the file is skipped entirely — no I/O, no task created.

Example: WHERE order_date = '2024-03-15' on a table with 1,000 files. If 950 files' date ranges don't include March 15, only 50 files are read. 95% I/O reduction.

Z-Ordering amplifies this: OPTIMIZE table ZORDER BY (customer_id, order_date) physically colocates rows with similar values in the same files — making min/max ranges tight, maximizing skip hit rate.

Traditional SQL B-Tree Index comparison:

  • B-Tree → row-level precision, O(log n) lookup, great for OLTP point queries. Requires random I/O to fetch scattered rows — catastrophic on object storage (ADLS) where random reads are expensive.
  • Delta Data Skipping → file-level granularity (less precise), but avoids all I/O for skipped files — perfect for analytical scans on object storage where sequential reads are cheap.
  • B-Tree requires maintenance (INSERT/UPDATE/DELETE updates the index). Delta stats are written automatically on each write — zero maintenance.

What is the impact of Broadcast Hints on worker memory? When should you avoid them?
MemoryJoin Strategy
▼

A broadcast hint sends a full copy of one DataFrame to every executor in the cluster. Memory impact = table size × number of executors.

Example: 500MB table, 200 executors → 100GB of total executor memory consumed just for that broadcast. If executors have 8GB each, this exceeds their capacity → OOM → executor crashes.

When to AVOID broadcasting:

  • Table is larger than ~200-400MB (depends on executor memory).
  • Cluster has many executors — even a modest-sized table becomes a memory problem at scale.
  • The "small" table has variable size — today it's 50MB, next month 2GB. Hardcoded broadcast hints become time bombs.
  • Already under OOM pressure — broadcasting adds fixed memory overhead regardless of the query.

Best practice: Rely on AQE's automatic broadcast decision (spark.sql.adaptive.enabled=true) which uses actual runtime statistics rather than your estimate. Only use explicit broadcast hints when you're certain the table fits and AQE's size estimate is wrong (e.g., it doesn't know about the filtered size after a predicate).

⚠️
Check spark.sql.autoBroadcastJoinThreshold (default 10MB). If you're using explicit broadcast() hint on a 200MB table and the default threshold is 10MB, you're overriding Spark's safety check intentionally — make sure your executors can handle it.

What is a Data Vault and how does it differ from Star Schema?
Advanced Modelling
▼

Data Vault is a modelling methodology designed for enterprise data warehouses where sources are many, change frequently, and history must be preserved. It has three entity types:

  • Hub: Stores unique business keys (e.g., hub_customer with customer_id, hash key, load date, source). No attributes — just the key and metadata.
  • Link: Stores relationships between hubs (e.g., link_order_customer connects hub_order and hub_customer). Many-to-many relationships handled naturally.
  • Satellite: Stores descriptive attributes and their history (e.g., sat_customer_details with name, address, load date). Full history maintained automatically — every change adds a new row.

Data Vault vs Star Schema:

  • Agility: Data Vault is additive — add new sources by adding new satellites, never modifying existing structure. Star Schema requires redesign when sources change.
  • Query simplicity: Star Schema is much simpler to query (few JOINs, business-friendly). Data Vault requires complex JOINs across Hubs, Links, and Satellites — analysts never query it directly. You build a Business Vault or Information Mart (star schema) on top.
  • Use case: Data Vault for complex enterprise DW with 50+ sources. Star Schema for straightforward BI reporting with well-understood sources.

How do you design a data model for a multi-tenant SaaS application?
ScenarioDesign
▼

Multi-tenant data modelling requires balancing isolation, cost, and query performance across tenants:

Option 1 — Shared database, shared schema (most common): All tenants in the same tables, with a tenant_id column everywhere. Cheapest, easiest to scale. Requires Row-Level Security to ensure tenants can't see each other's data. Risk: one large tenant's queries can impact others (noisy neighbor).

Option 2 — Shared database, separate schema: Each tenant gets their own schema (tenant_a.orders, tenant_b.orders). Good isolation, same database. Works well up to ~100 tenants. Gets unwieldy at thousands.

Option 3 — Separate database per tenant: Maximum isolation, compliance friendly (GDPR — delete one database = delete one tenant's data). Most expensive to maintain at scale.

In a Lakehouse context: Use ADLS folder structure: /tenants/{tenant_id}/silver/, /tenants/{tenant_id}/gold/. ADLS POSIX ACLs restrict each tenant's service principal to their folder. Shared Databricks cluster reads all tenants but filters by tenant_id at the query layer. Synapse Serverless SQL external tables use folder-level access control.

💡
Always include tenant_id in partition keys and distribution keys. Queries filtered by tenant_id will prune entire partitions/distributions — critical for performance when you have 10,000 tenants.

How would you model a time-series dataset for efficient querying?
ScenarioTime-Series
▼

Time-series data (IoT sensor readings, stock prices, website events) has specific patterns: always appended (rarely updated), queried by time range and entity, aggregated over windows.

Partitioning: Partition by date (or date + hour for high-frequency data). partitionBy("date") in Delta/Parquet means time-range queries read only relevant partitions — critical for performance on years of data.

Ordering within partitions: Sort within partitions by (entity_id, timestamp). Combined with data skipping, this allows per-entity range queries to read very few files.

Z-Ordering: OPTIMIZE sensor_readings ZORDER BY (sensor_id, timestamp) colocates readings for the same sensor in the same files. Queries like WHERE sensor_id = 'S123' AND timestamp BETWEEN X AND Y become extremely fast.

Aggregation tables: Pre-compute hourly, daily, monthly aggregates as separate Delta tables. BI tools query the pre-aggregated table — not the raw 1-trillion-row table.

Retention policies: Delete/archive data older than the retention window using Delta VACUUM + ADLS lifecycle policies. Don't let the table grow unboundedly.

  • Parallel effort: If there's a team, one person communicates with stakeholders while another debugs the technical issue.
  • After the demo: Full root cause analysis. Add a pre-demo checklist. Implement monitoring that would have caught this earlier.

Key message: "I would never let a technical issue become a surprise to the client. I communicate early, give options, and focus on what we CAN show rather than what's broken."

👥 Collaboration & Stakeholder Management

How do you collaborate with data analysts and business teams?
Collaboration
▼
What they're looking for: Cross-functional communication, empathy, translating technical concepts to business language.
  • Start with their language: Analysts think in metrics, business outcomes, and KPIs — not partitions and shuffle operations. Translate: "The pipeline now updates every hour instead of daily" not "I reduced the micro-batch interval."

How do you handle multiple stakeholders with conflicting requirements?
Stakeholder Management
▼
What they're looking for: Diplomatic communication, escalation path, principled decision-making, not just people-pleasing.
  • Understand the conflict: Is it a priority conflict (both want it first), a technical conflict (incompatible requirements), or a resource conflict (same team capacity)? Each needs a different solution.
  • Find the shared goal: Usually both stakeholders want the same business outcome. Start there — "Both of you want faster insights, let's see how to serve both."
  • Escalate with options, not problems: Don't go to the manager saying "they disagree." Go with: "Here are 3 options, with these tradeoffs. Which do you want me to pursue?"
  • Data informs priority: "Stakeholder A's requirement serves 500 users daily; Stakeholder B's serves 10. In the absence of other guidance, I'd prioritize A."

How do you explain a technical bottleneck to a non-technical client or stakeholder?
Communication
▼
What they're looking for: Communication clarity, empathy, business-impact framing — not technical jargon.
  • Lead with impact, not cause: Not "the shuffle partition size is causing executor OOM." Instead: "Our pipeline is taking 4 hours instead of 1 because it's processing more data than it was designed for."
  • Use analogies: "Think of it like a single checkout lane at a supermarket — when 1,000 customers show up at once, everyone waits. We need to open more lanes [parallelize the processing]."
  • Give options with tradeoffs in business terms: "Option A: Fix takes 2 days, costs ₹5000/month extra. Option B: Quick workaround takes 4 hours, works for the next 3 months." Let them choose with full information.
  • Give a timeline and set expectations: "I expect to have this resolved by Thursday. I'll update you daily at 10 AM. If anything changes, I'll reach out immediately."
  • Don't over-explain: Stop when you've communicated the impact, the cause in one sentence, the resolution plan, and the timeline. More details than that is usually noise to a non-technical stakeholder.

10
How do you ensure reliability and scalability in your data solutions?
Technical Behavioral
▼
What they're looking for: Engineering mindset — you build things that last, not just things that work once.
  • Reliability through idempotency: Every pipeline is designed to be safe to re-run without creating duplicates or data loss. MERGE operations, watermarks, partition overwrites.
  • Reliability through monitoring: Every pipeline has associated alerts. Failures are never silent. Row count checks, schema validation, and null checks run at every stage boundary.
  • Reliability through testing: Unit tests for transformation logic, integration tests against a staging environment, data quality assertions in production.
  • Scalability through configuration-driven design: Adding a new data source is a config change, not a code change. The framework handles 1 table and 1,000 tables the same way.
  • Scalability through partitioning and distribution: Partition strategies designed upfront based on query patterns. Hash distribution on join keys in Synapse. Z-ordering in Delta Lake.
  • Scalability through separation of concerns: Ingestion, transformation, and serving are independent — scaling one doesn't require changing the others.
11
How do you ensure data quality and reliability in your projects?
Technical Behavioral
▼
  • "Quality is not a QA step — it's built into every stage." I define data quality rules (completeness, uniqueness, validity, timeliness) at the beginning of a project, not after launch.
  • Contract with source teams: Define a data contract — expected schema, delivery frequency, SLA, and escalation path for breaches. When the source violates the contract, there's a clear process — not a surprised pipeline failure.
  • Automated quality checks at every layer: Bronze: schema check, row count > 0. Silver: null checks, uniqueness, referential integrity. Gold: business rule validation, reconciliation with source counts.
  • Observability: Every pipeline writes audit rows (run ID, table, rows in, rows out, start/end, status). Anomalies (count outside ±10% of 7-day average) trigger alerts before anyone notices in a dashboard.
  • Respond to issues quickly: I've configured Azure Monitor alerts that reach the on-call engineer within 5 minutes of pipeline failure — not 8 hours later when the business opens.
