Spark runs on a master-worker architecture. The Driver is your program — it builds the execution plan and coordinates everything. Executors are JVM processes on worker nodes that actually run the tasks and cache data.

🎯
Driver
Maintains SparkContext, builds the DAG, runs the DAG Scheduler and Task Scheduler. Your app's JVM process. Sends serialised tasks to executors.
⚙️
Executor
JVM on a worker node. Runs tasks in a thread pool, stores cached partitions in Block Manager, sends heartbeat every 10s to the Driver.
📐
DAG Scheduler
Converts RDD/DataFrame lineage into a DAG of stages. Splits on shuffle boundaries. Submits stage-ready TaskSets to the Task Scheduler.
🔀
Task = 1 Partition
One task processes exactly one partition on one executor thread. Parallelism = min(available cores, number of partitions).
✦ Lazy evaluation — the key insight

Transformations (filter, select, join) just build a logical plan — nothing runs. An action (count(), write(), collect()) triggers the entire chain. This lets Catalyst see the full plan upfront and optimize globally — predicate pushdown, column pruning, join reordering.

From .count() to results — the 7-step journey

  • 1
    Action triggers the job
    You call df.count(). This is an action. Without an action, the entire chain is just a recipe — nothing has executed. SparkContext kicks off a new Job.
  • 2
    DAG Scheduler builds the graph
    Traverses the RDD/DataFrame lineage backwards. Cuts it into Stages wherever a shuffle (wide dependency) is found. Stages with no pending parents are immediately submitted as TaskSets.
  • 3
    Catalyst optimizes (DataFrame/SQL only)
    Rewrites the logical plan: pushes filters to scan, prunes unused columns, reorders joins using cost-based stats. Outputs the best physical plan — SortMergeJoin vs BroadcastHashJoin, etc.
  • 4
    Task Scheduler assigns tasks to executors
    Scores each assignment by data locality: PROCESS_LOCAL (data in same JVM) → NODE_LOCAL (same machine) → RACK_LOCAL → ANY. Prefers sending the task to where data already lives.
  • 5
    Executor runs the task
    Deserializes the task, reads its partition, processes via Tungsten's whole-stage code generation (fused JVM bytecode — no virtual-call overhead), writes output to memory or shuffle files.
  • 6
    Shuffle (between stages)
    Reducers fetch map output from across the cluster. Sort-based shuffle writes sorted files per partition; executors pull only the ranges they own. Most expensive step — minimize shuffles.
  • 7
    Results returned to Driver
    After the final stage, results flow back. count() returns a single long. collect() returns all rows — dangerous on large datasets (Driver OOM risk).
# Annotated execution flow
df = spark.read.parquet("/mnt/data/sales/")       # lazy — adds scan node to DAG
df2 = df.filter("amount > 1000")               # lazy — filter pushed to scan by Catalyst
df3 = df2.groupBy("region").sum("amount")     # lazy — SHUFFLE BOUNDARY → new Stage
df3.write.format("delta").save("/mnt/gold/")   # ACTION → DAG executes now

# Stage 0: scan + filter (narrow) — runs 1 task per file partition
# Stage 1: reduceByKey (wide) — runs after shuffle, spark.sql.shuffle.partitions tasks
✦ Data locality scoring

PROCESS_LOCAL — data in same JVM (best). NODE_LOCAL — same physical node. RACK_LOCAL — same rack. ANY — different rack, network transfer (worst). Spark waits a configurable delay (spark.locality.wait, default 3s) before falling back to the next tier.

DAG — Directed Acyclic Graph

The DAG is Spark's execution plan. Narrow transformations (each partition feeds exactly one output partition) stay in the same stage. Wide transformations (each output partition may pull from many input partitions) force a shuffle boundary — a new stage begins after the shuffle.

JobTriggered by one action (count, write, collect)1 action = 1 job

TaskSmallest unit — 1 task = 1 partition on 1 executor threadParallelism = task count

⚠ Interview trap — cache() vs checkpoint()

cache() — keeps RDD in memory, lineage intact, data lost on executor failure (recomputed from lineage). persist(DISK_ONLY) — spills to disk, lineage intact. checkpoint() — writes to HDFS, cuts lineage. Use checkpoint for iterative algorithms (ML, graph) where recomputing long lineage from scratch would be expensive.

⚠ Interview trap — coalesce() vs repartition()

repartition(n) — full shuffle, can increase or decrease partitions, evenly distributed. coalesce(n) — no shuffle, only decreases, merges local partitions. Use coalesce before a write to reduce output file count without paying the network cost of a full shuffle.

Narrow vs Wide Transformations

TypeDefinitionShuffle?Examples
NarrowEach output partition depends on exactly one input partitionNomap, filter, select, withColumn, union, mapPartitions

⚠ The golden rule

Every shuffle writes data to disk and transfers it over the network. The goal of Spark optimization is almost always: reduce the number and size of shuffles. This mental model frames every performance answer at a senior level.

# Narrow — each partition processed independently, stays in Stage 0
df.filter("amount > 0")          # narrow
  .withColumn("tax", col("amount") * 0.18)  # narrow
  .select("id", "amount", "tax")   # narrow — all Stage 0

# Wide — triggers shuffle → new Stage
  .groupBy("region")                # WIDE → shuffle boundary
  .agg(sum("amount"), count("id"))  # runs in Stage 1 after shuffle
What is Lazy Evaluation and why does Spark use it?
+

Lazy evaluation means transformations are not executed until an action is called. Spark builds a DAG of all transformations first, then the Catalyst Optimizer can see the full plan and apply global optimizations — predicate pushdown, column pruning, join reordering — that wouldn't be possible if each transformation executed immediately. Without laziness, you'd get eager row-by-row processing with no global view.

# Lazy — no execution yet
df = spark.read.parquet("/data/").filter("id > 100").select("id", "name")
# Action — triggers the DAG
df.count()
What is the difference between a Job, Stage, and Task in Spark?
+

Job: triggered by one action (count, write). Stage: a set of transformations with no shuffle boundary — tasks in a stage can run in parallel. Task: the smallest unit — one task processes one partition. Rule: 1 job → N stages (split at shuffles) → M tasks per stage (one per partition). Parallelism = min(available executor cores, number of partitions).

What is Spark Session vs Spark Context?
+

SparkContext (Spark 1.x) was the entry point for RDD-based programming — one per JVM. SparkSession (Spark 2.x+) is the unified entry point that wraps SparkContext and also provides DataFrame/Dataset API, SQL context, and Hive context. In Databricks notebooks, spark is a pre-created SparkSession — you never instantiate it manually.

spark = SparkSession.builder \
    .appName("MyJob") \
    .config("spark.sql.shuffle.partitions", "400") \
    .getOrCreate()
# sc = spark.sparkContext  # access SparkContext if needed
If an executor dies mid-job, what does Spark do?
+

The Driver detects the missing heartbeat (after spark.network.timeout). It re-schedules the failed tasks on other executors. If the failed stage had shuffle output now lost, Spark re-runs the parent stage too. RDD lineage is what enables this — Spark always knows how to recompute any partition from its inputs.

Explain the Unified Memory Model — how does execution borrow from storage?
+

Pre-Spark 1.6: storage and execution pools were fixed sizes — a job with huge shuffles but no caching wasted all storage memory. Unified model: Execution and Storage share one pool. Execution can evict cached (Storage) partitions if it needs more room — the cached RDD is simply recomputed from lineage if needed again. Storage cannot evict Execution. The only guarantee: Storage retains at least storageFraction of the unified pool. This prevents OOM during large shuffles at the cost of cache eviction.

What is whole-stage code generation (WSCG) and why does it matter?
+

Traditional Volcano model: each operator calls next() on its child — one row at a time, huge virtual-call overhead. WSCG (Tungsten) fuses the entire pipeline (scan → filter → project → aggregate) into one compiled Java method. No virtual calls, data stays in CPU registers across operators, enabling CPU SIMD vectorization. Result: 2–10× speedup on CPU-bound queries. Verify with df.explain(mode="codegen").

Every DataFrame transformation in Spark is lazy — calling .filter(), .select(), .join() just adds nodes to the logical plan. Nothing executes. Only when an action (.count(), .collect(), .write(), .show()) is called does Spark run the plan.

✦ Why this is a performance win

Because Spark has the full plan before executing, the Catalyst Optimizer can push filters down closer to the data source (predicate pushdown), eliminate unnecessary columns (column pruning), and reorder joins for efficiency. An eager system would execute each step blindly with no global view.

# Basic explain — shows physical plan
df.filter("amount > 1000").groupBy("region").count().explain()

# Extended — shows logical + physical plan
df.explain("extended")

# Cost-based — shows statistics used by optimizer
df.explain("cost")

What is the difference between cache() and persist()? When do you use each?
+

cache() is a shortcut for persist(StorageLevel.MEMORY_AND_DISK). persist() gives you control over the storage level. Use when a DataFrame is referenced more than once in your DAG — without caching, Spark re-reads and recomputes it every time.

# Cache (MEMORY_AND_DISK — spills to disk if memory full)
df.cache()  # or df.persist()

# Serialized (smaller memory footprint, slightly slower to deserialize)
from pyspark import StorageLevel
df.persist(StorageLevel.MEMORY_AND_DISK_SER)

# Always unpersist when done
df.unpersist()

When NOT to cache: When a DF is only used once (wastes memory). When it's very large and you don't have enough cluster memory. When it changes every run (streaming).

Adaptive Query Execution (AQE) — Spark 3.x

AQE re-optimizes the query plan at runtime using actual data statistics collected during execution — not just estimates. Three key features:

  • Dynamically coalesces shuffle partitions: If 200 shuffle partitions are created but most are tiny (due to filter reducing data), AQE merges them into fewer, larger partitions automatically.
  • Dynamically switches join strategies: If one side of a SortMergeJoin turns out to be small after a filter, AQE can switch it to a BroadcastHashJoin at runtime — even though the plan was built assuming large tables.
  • Dynamically handles skew: Detects skewed partitions and splits them into sub-tasks for parallel processing.
# Enable AQE (default: true in Spark 3.2+)
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

Shuffle Compression (Financial Pipeline Question)

📌 From your interview doc — Q10

"A large Spark pipeline for a financial institution processes massive aggregations, causing heavy shuffles. Goal: compress shuffle files when memory is insufficient."

# Compress shuffle map output files
spark.conf.set("spark.shuffle.compress", "true")

# Compress data spilled to disk when memory is insufficient
spark.conf.set("spark.shuffle.spill.compress", "true")

# Codec: lz4 (default, fastest), snappy (balanced), zstd (best ratio)
spark.conf.set("spark.io.compression.codec", "lz4")

# Also tune shuffle partitions for large data
spark.conf.set("spark.sql.shuffle.partitions", "800")  # default 200 too low for big data

How do you find and eliminate data skew in Spark?
+

Detect: In Spark UI → Stages tab, if one task takes 10× longer than others — that's skew. The skewed partition has disproportionately more data (usually due to a high-frequency key value like "NULL" or "Unknown").

Fix with salting: Add a random prefix to the skewed key before joining, then remove it after.

from pyspark.sql.functions import col, concat, lit, rand, floor

# Salt factor = 10 buckets
SALT = 10

# Large (skewed) table: add random salt prefix
large_df = large_df.withColumn("salted_key",
    concat(floor(rand() * SALT).cast("string"), lit("_"), col("join_key")))

# Small table: explode with all salt values
from pyspark.sql.functions import array, explode
small_df = small_df.withColumn("salt_vals", array([lit(str(i)) for i in range(SALT)])) \
                   .withColumn("salted_key", concat(explode(col("salt_vals")), lit("_"), col("join_key"))) \
                   .drop("salt_vals")

result = large_df.join(small_df, "salted_key").drop("salted_key")
repartition() vs coalesce() — when to use which?
+

repartition(n): full shuffle, creates exactly n evenly distributed partitions. Can increase or decrease. Use when you need to increase parallelism or redistribute skewed data.

coalesce(n): no shuffle, merges existing partitions. Can only decrease. Faster but can create uneven partitions. Use when writing to fewer files — but be careful: coalesce(1) collapses everything to the driver, causing a bottleneck.

# After a filter that reduced data volume — fewer partitions needed
df_filtered = df.filter("status = 'ACTIVE'").coalesce(50)

# Before a join on a new key — redistribute evenly
df_repartitioned = df.repartition(200, "customer_id")

Broadcast Join — when NEVER to use it

🚫 Never broadcast in these cases
  • FULL OUTER JOIN: Spark does not support broadcast for full outer joins — silently falls back to SortMergeJoin.
  • "Small" table is actually large: A table with few rows but wide columns (BLOBs, arrays, nested structs) can be GBs in memory. Broadcasting it OOMs every executor.
  • Streaming joins: You cannot broadcast a streaming DataFrame.
  • LEFT ANTI / SEMI when right side is larger: Broadcasting the large side fills all executor memory.
from pyspark.sql.functions import broadcast

# Correct: explicitly broadcast verified-small table
result = large_df.join(broadcast(small_dim_df), "product_id", "inner")

# Disable auto-broadcast (prevent Spark from guessing wrong)
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "-1")

# Check what join was used:
result.explain()  # look for BroadcastHashJoin vs SortMergeJoin

Anti-Semi Join — TCS May 2024

A LEFT ANTI JOIN returns rows from the left table that have no match in the right table. A LEFT SEMI JOIN returns rows from the left that do have a match (but no columns from right).

# Find customers who have NEVER placed an order
no_orders = customers_df.join(orders_df, "customer_id", "left_anti")

# Find products that have at least 1 order (semi join)
ordered_products = products_df.join(orders_df, "product_id", "left_semi")

# SQL equivalent of anti-join:
spark.sql("""
  SELECT c.* FROM customers c
  WHERE NOT EXISTS (SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id)
""")
✦ Why anti-join over NOT IN

NOT IN silently returns no rows if the subquery contains any NULL values (three-valued logic). Anti-join handles NULLs correctly and is more efficient — it short-circuits on first match found.

Broadcast vs Hash Joins — which does not return NULL?
+

Neither join type inherently returns or avoids NULLs — that's determined by the join kind (INNER, LEFT, FULL), not the execution strategy (Broadcast vs SortMerge). INNER JOIN never returns unmatched rows so no NULL from join keys. LEFT JOIN returns NULLs for unmatched right-side columns. The execution strategy (Broadcast vs SortMerge) is purely a performance choice.

How does Spark decide between SortMergeJoin and BroadcastHashJoin?
+

Spark checks the estimated size of each side against spark.sql.autoBroadcastJoinThreshold (default 10MB). If either side is below this threshold, it auto-broadcasts. You can force broadcast with broadcast() hint. With AQE enabled, Spark can switch to BroadcastHashJoin at runtime even if the estimate was wrong — it re-evaluates after the smaller side finishes executing.

Q&A

How many partitions is right?

Rule of thumb: aim for partitions of 100–200 MB each. Too many small partitions = scheduling overhead. Too few large partitions = insufficient parallelism and potential OOM.

# Check current partition count
print(df.rdd.getNumPartitions())

# Estimate total data size, calculate target partitions
# e.g. 10 GB data → 10000/128 ≈ 80 partitions

# Repartition on a key for join optimization
df_partitioned = df.repartition(200, col("customer_id"))
# Now joining on customer_id won't need to shuffle — data is already co-located
✦ Is my data partitioned correctly for SQL queries?

If your most common query filters on date and your data is partitioned by date in Delta — Spark uses partition pruning and only reads the relevant date folders. Mismatched partition key = full table scan on every query. Always align partition key with the most common filter column.

How to handle aggregation partition skew

When grouping or joining on a skewed key (e.g., 80% of rows have region='UNKNOWN'), one partition becomes massive.

  • Option 1 — AQE skew handling: Enable spark.sql.adaptive.skewJoin.enabled=true. AQE automatically detects and splits large partitions.
  • Option 2 — Salting: Add a random prefix to the key before grouping, aggregate, then remove prefix and aggregate again.
  • Option 3 — Filter then union: Separate the skewed key, process it independently, union with the rest.
# Option 3: treat skewed key separately
df_normal = df.filter(col("region") != "UNKNOWN")
df_skewed = df.filter(col("region") == "UNKNOWN")

# Process skewed partition with higher parallelism
df_skewed_result = df_skewed.repartition(400).groupBy("region").sum("amount")
df_normal_result = df_normal.groupBy("region").sum("amount")

result = df_normal_result.union(df_skewed_result)
How can data transfers be kept to a minimum while using PySpark?
+
  • Predicate pushdown: Filter as early as possible — ideally at the scan level so less data is read from storage.
  • Column pruning: select() only the columns you need — avoids deserializing unused columns.
  • Broadcast small tables: Eliminates shuffle for one side of the join.
  • Cache intermediate results: Prevents recomputing the same data multiple times across actions.
  • Partition correctly: Co-locate join keys in the same partitions to avoid shuffle entirely.
  • Use Delta file skipping: Query filters map to Delta statistics — files that can't satisfy the filter are skipped at the reader level.

📌 Q3 from your Databricks doc

"How do you optimize Garbage Collection in Databricks?" — answer at 4 layers.

  • 1
    Use off-heap memory
    Move data storage outside the JVM heap. This dramatically reduces the volume of objects GC must manage.
  • 2
    Tune GC algorithm
    G1GC (default since Java 9) handles large heaps better than ParallelGC. Set -XX:InitiatingHeapOccupancyPercent=35 to trigger GC earlier and avoid long stop-the-world pauses.
  • 3
    Reduce object creation — avoid UDFs
    Python UDFs create one Python object per row, which the JVM must manage. Use native Spark functions (F.col, F.when, F.regexp_extract) — they operate on JVM column vectors with no per-row Python overhead.
  • 4
    Use serialized storage levels
    MEMORY_AND_DISK_SER stores cached data as compact byte arrays instead of deserialized objects. Fewer GC objects even at the cost of a small deserialization penalty at read time.

⚡ ACID Transactions

Every read/write on a Delta table is atomic. Partial writes never corrupt the table — if a job fails mid-way, Delta rolls back automatically using the transaction log. No more "half-written Parquet" incidents.

⏳ Time Travel

Query any historical version with VERSION AS OF or TIMESTAMP AS OF. Useful for debugging bad data loads, auditing, and rolling back accidental deletes — without maintaining separate backup tables.

📋 Schema Enforcement & Evolution

By default Delta rejects writes that don't match the table schema (schema enforcement). Opt-in to schema evolution with mergeSchema to safely add new columns without breaking existing queries.

🔀 Scalable Upserts (MERGE)

Native MERGE INTO support — the SQL standard upsert. Handles SCD Type 1 & 2, deduplication, and conditional deletes in a single atomic operation. No more multi-step overwrite patterns.

⚙ OPTIMIZE & Z-ORDER

OPTIMIZE compacts small files into larger ones (improving read performance). ZORDER BY co-locates related data so Spark can skip entire file ranges via data skipping — massive gains on high-cardinality filter columns.

🗑 VACUUM & File Management

VACUUM deletes old Parquet files no longer referenced by any live table version. Default retention is 7 days (168 hours). Running VACUUM below 7 days requires disabling the safety check — risky if concurrent readers exist.

Can I add a column in between existing columns in a Delta table?
›

Short answer: No — Delta (and Parquet) doesn't support inserting columns at a specific position.

Columns are always appended at the end with ALTER TABLE ... ADD COLUMNS. The physical Parquet files are column-by-position, so inserting mid-schema would require a full rewrite.

Workaround — use CREATE OR REPLACE VIEW to reorder logically:

CREATE OR REPLACE VIEW my_table_ordered AS
SELECT col1, new_col, col2, col3 FROM my_table;

If a new physical layout is truly needed, CREATE TABLE AS SELECT with the desired order, then rename/swap.

⚠ Interview question — Can you have 18 months of version history?

Short answer: Not by default (30 days), but yes with explicit configuration.

Two separate settings control this — most people confuse them:

-- Setting 1: How long Delta keeps old data FILES
-- (files referenced by old versions are not deleted before this)
ALTER TABLE my_table
SET TBLPROPERTIES ('delta.deletedFileRetentionDuration' = 'interval 540 days');

-- Setting 2: How long the transaction LOG entries are kept
-- (log compaction won't remove entries newer than this)
ALTER TABLE my_table
SET TBLPROPERTIES ('delta.logRetentionDuration' = 'interval 540 days');
-- 540 days ≈ 18 months

-- CRITICAL: You must also align VACUUM or it will delete the files!
-- VACUUM default retention = 168 hours (7 days) — this overrides your setting
VACUUM my_table RETAIN 12960 HOURS;  -- 540 days × 24 = 12,960 hours

-- Or disable auto-vacuum on this specific table
ALTER TABLE my_table
SET TBLPROPERTIES ('delta.autoOptimize.autoVacuum' = 'false');
Version history timeline — what's kept vs deleted
Day 0 — Table created, version 0
Initial data files written. Transaction log entry v0 created. Both data files and log entry kept indefinitely until VACUUM/log cleanup.
Day 1–30 — Normal writes (default retention)
Each write creates new Parquet files + log entry. Old files (replaced by newer versions) are kept for 7 days by default before VACUUM can delete them. Time travel available for all 30 days by default.
Day 31 — Default: history starts disappearing
Log entries older than 30 days are eligible for removal. Data files older than 7 days already deleted by regular VACUUM runs. Time travel to Day 1 no longer possible.
Day 31–540 — With 18-month config set
Both logRetentionDuration and deletedFileRetentionDuration set to 540 days. VACUUM called with RETAIN 12960 HOURS. All versions and their files remain accessible. Full time travel available.
⚠ Storage cost consideration
18 months of history = 18 months of old Parquet files. For a table with 1 GB of daily changes: 540 × 1 GB = ~540 GB extra storage just for history. Justify this cost — typically only for compliance/audit tables.
Common mistake
Set logRetentionDuration = 540 days but forget to update VACUUM retention. Next VACUUM run deletes all files older than 7 days — versions exist in the log but the underlying files are gone. Time travel throws FileNotFoundException.
Production pattern
Set both properties on table creation for audit tables. Disable auto-vacuum on those specific tables. Run a separate scheduled VACUUM job with the matching RETAIN window. Apply only to tables that need it — not globally.

+

mergeSchema=true: New columns in the incoming data are added to the table schema. Existing rows get NULL for new columns. Safe — no data loss.

overwriteSchema=true: Replaces the entire schema with the incoming data's schema. Used with mode("overwrite"). Destructive — use only when you intentionally want to change the table structure.

df.write.format("delta").mode("append") \
        .option("mergeSchema", "true").save("/path/")

df.write.format("delta").mode("overwrite") \
        .option("overwriteSchema", "true").save("/path/")

Star schema vs Snowflake schema — which does Delta prefer?
+

Star schema is strongly preferred for analytics on Delta/Databricks. Denormalized dimensions mean fewer joins, simpler DAGs, and better use of Spark's columnar execution. Snowflake schema introduces more joins which create shuffle boundaries. In Databricks, the recommendation is to denormalize dimension tables during the Silver → Gold transformation and keep Gold as a wide star schema ready for consumption.

Can you insert a column in between existing columns in a Delta table?
+

No. Delta Lake always appends new columns at the end of the schema. Workaround: create a view or CTAS with explicit column ordering.

CREATE OR REPLACE VIEW my_table_ordered AS
SELECT col1, new_col, col2, col3 FROM my_table;

# Key insight: highest product is either (2 largest positives) or (2 most negative)
# because negative × negative = positive

def max_product_pair(arr):
    if len(arr) < 2:
        return "Need at least 2 elements"
    sorted_arr = sorted(arr)
    # Candidate 1: two largest positives
    pair_high = (sorted_arr[-1], sorted_arr[-2])
    # Candidate 2: two most negative (their product is positive)
    pair_low  = (sorted_arr[0], sorted_arr[1])
    return pair_high if pair_high[0]*pair_high[1] >= pair_low[0]*pair_low[1] else pair_low

Overview & the DLT → SDP Rebrand

What changed, what stayed the same, and why it matters for your interview

2025
Major rebrand: Delta Live Tables (DLT) is now Lakeflow Spark Declarative Pipelines (SDP). At Data+AI Summit 2025, Databricks contributed SDP to the Apache Spark open-source project (Spark 4.1+). The import dlt module is replaced by from pyspark import pipelines as dp. No migration needed — all existing DLT code runs unchanged.
What is Lakeflow SDP?
Declarative ETL Framework

Lakeflow SDP is a declarative framework for building batch and streaming data pipelines in SQL and Python. You declare what the data should look like; SDP handles how to execute, orchestrate, retry, and govern it.

It runs on the performance-optimised Databricks Runtime and shares the same pyspark.pipelines module as open-source Apache Spark 4.1 — meaning your pipeline code is now portable to open-source Spark.

▶ How does "declarative" differ from imperative Spark code?

Imperative (plain Spark): You write step-by-step instructions — read this stream, apply this transformation, write to this location, manage checkpoints, handle retries, set up the orchestration order yourself.

Declarative (SDP): You write what the output should be. You define a function that returns a DataFrame — SDP determines when to run it, in what order, how to checkpoint it, how to retry failures, and how to parallelize independent steps.

Concrete analogy

SQL is declarative — SELECT * FROM orders WHERE status = 'ACTIVE' doesn't say how to scan, join, or filter. SDP applies the same philosophy to entire pipelines. You write transformation logic; the engine builds the execution plan.

Core component: Flows
New in SDP

SDP introduces the concept of a Flow — the foundational data-processing unit. A flow reads from a source, applies logic, and writes to a target. Flows replace the implicit "table as source" model of DLT.

Flow types: Append (streaming, currently the only exposed type), AUTO CDC (streaming CDC with out-of-order handling), and Materialized View (batch incremental).

▶ Explain the retry hierarchy in SDP

SDP implements a 3-level retry hierarchy, always starting with the cheapest unit:

Level 1 — Spark Task retry

If a single Spark task fails (e.g. one partition executor crashes), Spark retries just that task on another node. This is the cheapest recovery — no reprocessing of other tasks.

Level 2 — Flow retry

If task-level retries are exhausted, SDP retries the entire flow (e.g. re-reads the micro-batch and re-applies the transformation). Checkpoint is rewound to the last committed offset.

Level 3 — Pipeline retry

If flow-level retries fail, SDP restarts the entire pipeline. For continuous pipelines, this means re-establishing all streaming queries and replaying uncommitted offsets.

@dlt.table decorator

This is a Python decorator that registers the function as a pipeline dataset. DLT/SDP scans all notebook cells during planning phase and collects every decorated function to build the DAG. The function itself is not executed at decoration time — it's executed when the pipeline runs that flow.

table_properties — pipelines.reset.allowed: false

This property prevents the pipeline UI from offering a "full refresh" on this table. Critical for large production Bronze tables where a reset means reprocessing months of data and incurring massive cost. Set this in production always.

spark.readStream.format("cloudFiles")

cloudFiles.schemaLocation

Auto Loader stores the inferred schema at this path separately from the streaming checkpoint. This is why on a pipeline reset (which clears checkpoints), the schema persists. You must manually clear the schemaLocation path if you want full schema re-inference.

_metadata.file_path

A hidden metadata column exposed by Auto Loader (and Spark's file sources generally) that gives you the full path of the source file for each row. Critical for auditability and debugging — you can trace any row back to its source file.

▶ dlt.read() vs dlt.read_stream() — critical distinction

dlt.read("table_name")

Reads the entire current contents of a pipeline dataset as a batch DataFrame. Use this in Materialized Views that need to aggregate or join all data. Under the hood, SDP's incremental MV engine optimises this to only reprocess changed data — but from your code's perspective, you write as if it's a full read.

dlt.read_stream("table_name")

Returns a streaming DataFrame for incremental processing. Only reads new rows appended since the last checkpoint. Required in Streaming Tables. Cannot be used in Materialized Views.

The gotcha

If you accidentally call dlt.read_stream() on a table that is the target of APPLY CHANGES INTO (SCD Type 2), you'll get a pipeline error — those tables cannot be streamed from within the same pipeline. Use dlt.read() instead.

⚠
Interview trap: "Can you use a Materialized View as a streaming source with dlt.read_stream()?" — No. MV targets are batch-computed. You can only stream from Streaming Tables (append-only sources). If you need to stream from an MV, read it as batch and apply incremental logic via APPLY CHANGES.

▶ Deep dive: sequence_by and out-of-order event handling

The problem: late-arriving events

In a Kafka CDC stream from Debezium, it's common for a DELETE event to arrive before the final UPDATE event (network reordering, partition lag differences). Without ordering logic, you'd apply the delete and then mistakenly restore the row with the late-arriving update.

How sequence_by fixes this

sequence_by tells SDP to use the value of this column (typically a timestamp or sequence number from the source DB) as the authoritative ordering for events on the same key. SDP buffers events and applies them in sequence_by order, not in arrival order.

Under the hood, SDP uses a MERGE INTO statement with a WHEN MATCHED AND sequence > current_sequence condition. It only applies the change if the incoming event's sequence is higher than what's already stored.

ignore_null_updates

False (default): a null value in the incoming CDC event explicitly overwrites the existing non-null value in the target. Use this when nullification is a valid business operation.

True: null values in the incoming event are ignored — existing non-null target values are preserved. Use when your CDC feed emits partial updates (only changed columns, nulls for unchanged).

▶ What SDP generates under the hood for SCD Type 2

The internal MERGE logic

When a CDC UPDATE arrives for customer_id = 42, SDP executes a 2-step operation atomically:

Step 1: UPDATE the current row — set __END_AT = new_sequence_value, __CURRENT = false

Step 2: INSERT the new version row — all columns from the incoming event, __START_AT = new_sequence_value, __END_AT = NULL, __CURRENT = true

This is a Delta Lake MERGE operation, which means it is ACID — no race condition between the close of the old version and the creation of the new version.

track_history_column_list

Without this, every single column change (even a last_login timestamp that changes every session) creates a new history row, bloating the table. By specifying track_history_column_list, you only version-track the columns that matter for slowly-changing dim logic. Other column changes are applied as SCD Type 1 overwrites on the current row.

The SCD2 streaming source restriction

A table created via apply_changes(stored_as_scd_type="2") cannot be used as a streaming source (dlt.read_stream()) within the same pipeline. You can read it as batch (dlt.read()). This is because SCD2 tables are internally backed by a Delta table with both current and historical rows — streaming from them would deliver both inserts and the "close-previous-version" updates, which breaks most streaming consumers.

Debezium integration pattern: Debezium Kafka records use op values of "c" (create), "u" (update), "d" (delete), "r" (read/snapshot). Map these: apply_as_deletes=expr("op = 'd'"). Use sequence_by=col("ts_ms") (the Debezium transaction timestamp in milliseconds).

▶ Why Enhanced Autoscaling is critical for streaming — technical internals

Standard autoscaling failure mode

Standard Spark autoscaling monitors executor CPU and memory. In a streaming pipeline, during a low-traffic period (e.g. 3am), the pipeline processes small batches quickly. CPU is low → autoscaler scales down to minimum workers. Then at 8am, Kafka lag spikes to 50M records. The autoscaler reacts to rising CPU, but by the time CPU climbs, tasks are already queueing. Scale-up lags behind the demand spike by 2-5 minutes.

Enhanced Autoscaling solution

Enhanced Autoscaling monitors the Kafka consumer group lag (or Auto Loader file queue depth) before it translates to CPU pressure. When lag exceeds a threshold, workers are added proactively. By the time the heavy batch starts processing, capacity is already present.

Under the hood, Databricks' Enhanced Autoscaling daemon polls the Kafka offset lag every ~30 seconds. It calculates an "estimated processing time at current worker count" and scales to meet a target lag SLA.

Interview answer

Enhanced = predictive, demand-signal-based. Standard = reactive, utilisation-based. For streaming, always use Enhanced — standard autoscaling was designed for batch workloads where latency is less critical.

▶ _rescued_data — what it contains and how to use it

What is _rescued_data?

When Auto Loader encounters a JSON/CSV record containing a column that doesn't exist in the current inferred schema, it doesn't drop the column or fail. Instead, it serialises the unknown column(s) as a JSON string in a special column called _rescued_data.

Example

Schema has: order_id, amount, status. Incoming record has: order_id, amount, status, promo_code (new column). promo_code is captured in _rescued_data as {"promo_code": "SAVE20"}.

Internals — DAG Planning, Checkpointing, Execution

What separates Senior from Staff-level answers in an SDP interview

Two-phase execution model

Phase 1 — Planning: SDP scans all source files in the pipeline. It imports all Python functions decorated with @dlt.table / @dp.table / etc., resolves all dlt.read() and dlt.read_stream() calls, and builds a directed acyclic graph (DAG) of dependencies. Tables with no inter-dependencies get scheduled for parallel execution.

Important: During planning, SDP evaluates your Python code. This means any Python logic outside the function body (module-level code) runs during planning. Avoid side effects — put all logic inside the decorated function.

Phase 2 — Execution: SDP submits Spark jobs per pipeline node in DAG order. Streaming nodes run as persistent Structured Streaming queries. Batch MV nodes run as Spark batch jobs. The execution engine writes progress events to the event log after each microbatch.

▶ What happens when you update pipeline code — incremental vs full refresh

On pipeline code update (no reset)

SDP diffs the new DAG against the previous one. Unchanged streaming tables resume from their existing checkpoints. Changed tables: SDP checks if the change is compatible (adding a column, adding a filter) or incompatible (adding a stateful aggregation, changing a key column in APPLY CHANGES). Incompatible changes force a checkpoint wipe for that table only — not the entire pipeline.

On full refresh (reset)

All checkpoints are deleted. All streaming tables restart from offset 0 (or earliest available). Materialized views are fully recomputed. The schemaLocation is NOT cleared — you must do this manually if you need fresh schema inference.

Selective refresh (update mode)

You can refresh specific tables without resetting the whole pipeline using the API: POST /api/2.0/pipelines/{pipeline_id}/updates with {"full_refresh_selection": ["table_name"]}. Available in Triggered mode only.

▶ Exactly-once semantics — how DLT achieves it

The challenge

Spark Structured Streaming provides exactly-once guarantees via checkpointing — each microbatch commits its output offset and writes data atomically. But a pipeline restart could re-process the last uncommitted microbatch, potentially writing duplicate data.

How SDP achieves exactly-once end-to-end

SDP combines two mechanisms:

1. Delta Lake ACID writes: Each microbatch writes to Delta as a single atomic transaction. The Delta transaction log records the exact files written for each batch. On replay, SDP uses idempotent writes — if the same batch ID is re-submitted, the duplicate write is a no-op.

2. Checkpoint + Delta log coordination: The streaming checkpoint records the last committed offset. The Delta transaction log records which data was written for that offset. On restart, SDP reads both — if checkpoint says "batch 47 committed" but Delta log shows batch 47 files were not yet written, it replays batch 47 idempotently.

Result: Even if the driver crashes mid-write, on recovery, SDP replays the uncommitted batch and the Delta idempotency guard prevents double-writes.

Q1 What is the difference between a Streaming Table and a Materialized View in DLT/SDP? ▼

A Streaming Table reads data incrementally from an append-only source (Kafka, Auto Loader) and maintains a Spark Structured Streaming checkpoint. Each pipeline run processes only new data since the last committed offset. It cannot handle updates or deletes on source data natively — use AUTO CDC for that.

A Materialized View applies transformation logic (joins, aggregations) and persists the result as a Delta table. SDP's incremental MV engine detects which upstream data changed and reprocesses only the delta — but from the developer's perspective, you write batch semantics. MVs can read from tables that have had updates/deletes. They do NOT maintain a streaming checkpoint.

Key differentiator: Streaming Table = streaming source + checkpoint state. MV = batch logic + SDP-managed incremental engine. Use SLT for Bronze ingestion; MV for Silver/Gold joins and aggregations.
Q2 What are the three expectation types? When would you use expect_or_fail, and what's the risk? ▼

@dlt.expect — warn and track violations in the event log; row is kept. Use for monitoring SLAs without blocking the pipeline.

@dlt.expect_or_drop — drop violating rows silently; tracked in event log. Use for Silver/Gold where downstream needs clean data but pipeline continuity matters more than row completeness.

@dlt.expect_or_fail — fail the entire pipeline on any violation. Use for zero-tolerance scenarios: financial reconciliation, regulatory reporting, data with contractual SLAs where bad data is worse than pipeline downtime.

The risk of expect_or_fail on streaming: In a Continuous streaming pipeline, a single malformed Kafka message triggers a pipeline failure, halting ALL downstream tables until the pipeline is manually restarted. This is rarely acceptable in production — instead, use expect_or_drop + a quarantine table + monitoring alert on the quarantine table row count.
Q3 How does DLT manage pipeline dependencies and execution order? ▼

DLT scans all source notebooks/files during a planning phase and collects every decorated function (@dlt.table, @dlt.view, etc.). It resolves all dlt.read() and dlt.read_stream() calls to build a directed acyclic graph (DAG) of dependencies. Tables at the same DAG level with no inter-dependencies execute in parallel. Tables with dependencies execute in topological order.

Key point: You never need to specify execution order. Just use dlt.read("table_name") and DLT infers the dependency automatically. This replaces the manual ADF dependency wiring or Airflow task ordering you'd need with plain Spark.
Q4 What is _rescued_data and when does it appear? ▼

_rescued_data is a system column automatically added by Auto Loader (and DLT's rescue schema mode) when incoming data contains columns not present in the current schema. Instead of dropping the unknown column or failing, the column's value is serialised as JSON into _rescued_data.

It appears when: (1) a new column is added to the source system schema, (2) a source sends an unexpected field, or (3) a column name has a case mismatch (schema is case-insensitive by default).

Production best practice: Add @dlt.expect("no_schema_drift", "_rescued_data IS NULL") to your Bronze table as a warning. This gives you a pipeline-level alert when source schema changes without halting ingestion. Then review the rescued data to decide whether to evolve the schema.
Q5 What is the new pyspark.pipelines API and what changed from import dlt? ▼

In 2025, Databricks contributed DLT to Apache Spark as open-source Spark Declarative Pipelines, available in Spark 4.1+ via the pyspark.pipelines module. The DLT-proprietary import dlt is now replaced by from pyspark import pipelines as dp.

Key API changes: @dlt.table (streaming) → @dp.table; @dlt.table (batch) → @dp.materialized_view (explicit separation); @dlt.view → @dp.temporary_view. All read functions keep the same names. AUTO CDC and dp.apply_changes() remain Lakeflow-exclusive — not in open-source Spark.

Migration note: No migration needed — old import dlt code runs unchanged. Migrate only if you want open-source Spark 4.1 portability.
Q6 Explain Triggered vs Continuous pipeline mode and when you'd choose each. ▼

Triggered: Pipeline starts on a schedule (or manual trigger), processes all available data, then the cluster terminates. Latency: minutes (cluster spin-up + processing time). Cost: pay only per run. Best for nightly ETL, hourly batch reporting pipelines, scenarios where near-real-time is not required.

Continuous: Pipeline runs perpetually on an always-on cluster. Processes data in microbatches (seconds to sub-minute latency). Uses Enhanced Autoscaling (backlog-based). Cost: continuous DBU burn even during idle periods. Best for real-time dashboards, CDC replication, IoT telemetry, SLA-sensitive pipelines.

Decision framework: If business requires data <5 minutes fresh → Continuous. If hourly refresh acceptable → Triggered. If variable load with unpredictable spikes → Serverless.
Q7 Design a DLT pipeline for CDC from Oracle via Debezium/Kafka, maintaining SCD Type 2 in Silver. List the gotchas. ▼

Architecture: Oracle → Debezium → Kafka → Bronze Streaming Table (raw CDC) → APPLY CHANGES INTO → Silver SCD2 Streaming Table.

Key implementation details: (1) Bronze table reads from Kafka with readStream.format("kafka") and parses Debezium payload. (2) apply_changes() with stored_as_scd_type="2", sequence_by=col("ts_ms") (Debezium's transaction timestamp), apply_as_deletes=expr("op = 'd'"). (3) Map Debezium ops: c/r=insert, u=update, d=delete.

Gotchas: (1) The SCD2 target table cannot be used as a streaming source — must use dlt.read(). (2) Debezium snapshot events have op="r" — treat as inserts. (3) Schema changes in Oracle propagate to Debezium payload — rescue mode on Bronze is critical. (4) ts_ms in Debezium is database transaction timestamp, not Kafka message timestamp — critical for correct ordering. (5) If Debezium sends partial column updates (only changed fields, nulls for unchanged), set ignore_null_updates=True.

Bonus point: After Silver SCD2, query current state with WHERE __CURRENT = true for downstream Gold aggregations. Use __START_AT / __END_AT for point-in-time reporting — great for GDPR data erasure tracking.
Q8 How would you implement a quarantine pattern in DLT? Why two tables instead of a flag column? ▼

Two tables pattern: one Silver table with @dlt.expect_all_or_drop(rules_dict) that receives only clean rows; one quarantine table that reads from the same Bronze source and filters to rows that fail any rule (inverse of the rules combined with OR).

Why not a flag column: Downstream consumers of Silver would need to filter WHERE dq_passed = true — this filter is easily forgotten, and a missed filter in a BI report propagates bad data silently. Separate tables enforce the quality boundary architecturally. The quarantine table can have different retention policy, be monitored by a separate DQ team, and be re-processed once the upstream issue is fixed.

Monitoring: Set an alert: "if quarantine table receives more than N rows in a pipeline run → PagerDuty alert to data engineering on-call."

Q9 A DLT streaming pipeline intermittently misses files. What do you investigate? ▼

Investigation checklist:

1. Checkpoint integrity: Is the checkpoint directory on stable storage (not a frequently-purged temp location)? A corrupted or partial checkpoint causes Auto Loader to restart the file index scan from a wrong offset.

2. cloudFiles.maxFilesPerTrigger: If set too low (default: 1000), files arriving faster than this limit will be batched across runs. Check event log for num_output_files — if consistently hitting the limit, increase it or switch to cloudFiles.maxBytesPerTrigger.

3. File notification vs directory listing mode: Auto Loader defaults to directory listing for small-scale. For production ADLS, enable Event Grid file notifications (cloudFiles.useNotifications=true) — listing has eventual consistency issues at high file volumes.

4. S3/ADLS eventual consistency: File written by an upstream process may not be immediately visible to the listing. Add a delay or use event notification mode.

5. Schema location corruption: If schemaLocation has a stale/corrupt schema, files with a newer schema might be skipped. Clear and regenerate schemaLocation.

Q13 When would you NOT use DLT? What are its limitations? ▼

Limitations / when to avoid:

1. Arbitrary Spark code in decorators: SDP evaluates decorated functions multiple times during planning. Side effects (file writes, external API calls, arbitrary database queries) inside decorated functions produce unpredictable results.

2. Non-Delta targets: DLT only writes to Delta Lake tables. If you need to write to JDBC, Kafka, or external systems as the primary output → use plain Spark with foreachBatch.

3. Complex event-driven logic: If your pipeline needs to react to pipeline events (e.g. "run Gold only if Silver had >0 rows") → use Lakeflow Jobs with conditional task logic instead.

4. Vendor lock-in risk: While pyspark.pipelines is now open-source, APPLY CHANGES INTO / AUTO CDC is Databricks-exclusive. If multi-cloud portability is a hard requirement, evaluate using Apache Spark SDP without CDC.

5. Premium plan required: DLT/SDP requires Databricks Premium — not available on Standard tier workspaces.

Q14 A 500GB+ Streaming Table needs a full pipeline reset. What are the implications and how do you minimize downtime? ▼

Implications: Full reset wipes checkpoints for ALL streaming tables in the pipeline, then triggers a full recompute from source. For a 500GB Bronze table from ADLS, this means re-reading every file ever ingested. Cost: significant compute cost, time: hours to days depending on source volume.

Mitigation strategy:

1. Prevent accidental resets: Set table_properties={"pipelines.reset.allowed": "false"} on large production tables. This blocks the UI reset button for that table.

2. CLONE before reset: CREATE TABLE bronze_orders_backup CLONE bronze_orders — shallow clone preserves Delta log without copying data. Rollback possible if reset goes wrong.

3. Selective refresh: Use the REST API with "full_refresh_selection": ["only_the_changed_table"] to reset just the affected table, not the whole pipeline.

4. Backfill strategy: Instead of full reset, write a separate one-time Spark job to backfill historical data into the Delta table, then start the DLT pipeline only from the current date. Use Delta's COPY INTO for idempotent historical backfill.

5. Zero-downtime pattern: Use table cloning + rename strategy: reset to new table name, backfill in background, atomic rename when ready — downstream consumers see no interruption.

Q15 Explain Enhanced Autoscaling vs standard Spark autoscaling. Why does the difference matter specifically for streaming pipelines? ▼

Standard autoscaling: Monitors executor CPU and memory. Adds workers when CPU exceeds threshold (typically >80% for N minutes). In streaming, during a low-traffic window the cluster scales down. When a spike arrives (e.g. Kafka backlog from a source restart), CPU doesn't rise until tasks start queuing — scale-up lags the demand by 2-5 minutes. During those 5 minutes, Kafka lag grows and streaming latency spikes.

Enhanced Autoscaling: Monitors the Kafka consumer group lag (or Auto Loader file queue depth) directly — a proxy for future CPU demand, not current CPU usage. Databricks' Enhanced Autoscaling daemon polls Kafka offset lag every ~30 seconds. When lag exceeds a threshold, it adds workers before CPU peaks. By the time the heavy batch starts processing, capacity is already available.

Why it matters: For a streaming pipeline with a latency SLA of <5 minutes, standard autoscaling routinely violates the SLA during demand spikes. Enhanced Autoscaling maintains consistent low latency by reacting to demand signals (backlog) rather than consequences (CPU saturation). Always use "mode": "ENHANCED" in Continuous pipeline cluster config.

Q16 How does SDP achieve exactly-once semantics end-to-end? What breaks it? ▼

Mechanism: SDP combines Spark Structured Streaming's checkpoint-based offset tracking with Delta Lake's ACID transactional writes. Each microbatch has a unique batch ID. Delta's write API is idempotent — if batch ID 47 is replayed after a crash, the duplicate write is detected via the Delta transaction log and no-oped.

What breaks exactly-once:

1. foreachBatch with non-Delta targets: If you use foreachBatch to write to JDBC, the write is not idempotent — a crash and replay causes duplicate rows in JDBC.

2. External side effects in decorators: An HTTP call or file write inside a DLT table definition runs multiple times (planning + execution). Not idempotent → duplicate effects.

3. Checkpoint on volatile storage: If the checkpoint directory is on a path that gets cleaned up (temp DBFS paths, ephemeral containers), the offset state is lost → pipeline replays from earliest offset → duplicates.

4. Manual Delta table overwrites during pipeline run: If you run TRUNCATE TABLE or INSERT OVERWRITE on a DLT-managed table while the pipeline is running, you corrupt the Delta transaction log's expected state → undefined behaviour.

Auto Loader is the workhorse of file-based ingestion on Databricks. The idea is simple: point it at a cloud storage prefix and it continuously picks up new files as they arrive — without ever re-reading old ones. It runs as a Structured Streaming query using cloudFiles as the source format, which means all the standard streaming machinery (checkpoints, restarts, backpressure) just works out of the box.

It has two detection modes. Directory listing is the default — Databricks periodically scans your prefix. Works fine for smaller workloads but slows down badly once you hit millions of files, because it has to list everything to find what's new. File notifications mode is what you want in production — it wires up cloud-native event queues (Azure Event Grid on Azure, SQS on AWS) so new file arrivals are pushed to Databricks in near real-time with zero listing overhead. Enable it with cloudFiles.useNotifications=true.

The killer feature is exactly-once processing. Auto Loader maintains a RocksDB-backed checkpoint that tracks every file it has processed. Cluster restart mid-run? It picks up exactly where it left off — no duplicates, no missed files. That's what makes it safe to land raw files straight into a bronze Delta table without any downstream deduplication logic.

  .trigger(availableNow=True)  # batch-mode: process all new files then stop
  .toTable("catalog.schema.events_bronze")
)
💡 Use availableNow=True in Jobs
This is probably the most underused trick. trigger(availableNow=True) tells Spark to process all currently available new files and then stop — like a batch job, but with streaming's exactly-once guarantees. You get the correctness of streaming at the cost of batch. Way cheaper than running an always-on cluster for a pipeline that only needs to run every 15 minutes.
⚠️ Never move the checkpoint location
Auto Loader ties its entire file-tracking state to the checkpoint path. If you move or delete it, Databricks has no memory of what it already processed — it'll re-read every file from the beginning and flood your bronze table with duplicates. Treat the checkpoint path as sacred. If you genuinely need to reset, do it intentionally and make sure your downstream tables can handle a full reload.
🚨 Directory listing on a huge prefix is a trap
This catches people all the time. If your storage prefix has 10M+ files and you're using the default listing mode, each run has to scan the whole prefix to find what's new. It gets progressively slower until your trigger interval can't keep up. The fix is cloudFiles.useNotifications=true — one config change, and now file arrivals are pushed via Event Grid / SQS instead of polled. Do this before you hit the wall, not after.
📌 Running Auto Loader from ADF
If your orchestration is still in ADF (common in enterprise Azure setups), wrap the Auto Loader notebook in an ADF Databricks Notebook activity and pass source_path and checkpoint_path as parameters. This gives you a reusable, parameterised ingestion notebook you can point at any landing zone without duplicating code.

Structured Streaming ● Mature GA
Apache Spark's native continuous stream processing engine
Overview
Patterns
Tips & Gotchas

Structured Streaming is the stream processing engine that everything else in Databricks is built on top of. Auto Loader uses it. Lakeflow Declarative Pipelines runs on it. When you consume from Kafka, Event Hubs, or Kinesis, you're writing a Structured Streaming job. The core abstraction is elegant: you write a DataFrame transformation as if it's a batch query, and Spark executes it incrementally on an infinite input stream. No RDD code, no low-level partition management.

The default execution model is micro-batch — Spark collects data for a configurable interval (say, 10 seconds), processes it as a mini-batch, commits the output and checkpoint, then starts the next one. This gives you good throughput and simple failure recovery, but latency is bounded by your batch interval. For lower-latency needs, Databricks added Real-Time Mode for Apache Spark (Data+AI Summit 2025) which uses a continuous processing model to get sub-second latencies — at the cost of slightly higher operational complexity.

The state management story is also important. For stateful operations — windowed aggregations, stream-stream joins, deduplication — Spark needs to store intermediate state somewhere. By default it uses an in-memory state store, which doesn't scale well. In production, always switch to RocksDB state store which spills to disk and handles much larger state efficiently.

# Structured Streaming — Kafka → Delta with Windowing
from pyspark.sql.functions import window, from_json, col
from pyspark.sql.types import StructType, StringType, TimestampType

schema = (StructType()
  .add("user_id", StringType())
  .add("event", StringType())
  .add("event_time", TimestampType())
)

raw = (spark.readStream
  .format("kafka")
  .option("kafka.bootstrap.servers", "broker:9092")
  .option("subscribe", "clickstream")
  .load()
  .select(from_json(col("value").cast("string"), schema).alias("data"))
  .select("data.*")
)

# 5-min tumbling window with 10-min watermark for late data
agg = (raw
  .withWatermark("event_time", "10 minutes")
  .groupBy(window("event_time", "5 minutes"), "event")
  .count()
)

(agg.writeStream
  .format("delta")
  .outputMode("append")
  .option("checkpointLocation", "/mnt/ckpt/clicks_window")
  .trigger(processingTime="30 seconds")
  .toTable("analytics.silver.click_windows")
)
💡 Enable RocksDB state store in production
This is one config line that makes a massive difference for stateful streaming: spark.conf.set("spark.sql.streaming.stateStore.providerClass", "com.databricks.sql.streaming.state.RocksDBStateStoreProvider"). The default in-memory state store runs out of memory and causes OOM restarts once your state grows beyond a few GB. RocksDB spills to disk transparently. Enable it from day one on any stateful job — windowed aggregations, stream-stream joins, deduplication — not after you hit a memory wall in production.
🚨 ADF SetVariable inside parallel ForEach — this will silently corrupt your data
SetVariable is not thread-safe. If you have a ForEach activity with parallelism > 1 and you're using SetVariable inside the loop body, multiple iterations will clobber each other's values. We've debugged this exact bug — the symptom is intermittently wrong variable values that are nearly impossible to reproduce consistently. The fix is to replace every SetVariable inside a parallel loop with AppendVariable. This is also why the return value of the copied_file_flag from child pipelines wasn't being read correctly in the analytics lab pipeline.
⚠️ Schema evolution requires a stream restart
If your upstream data source adds a new column and you want it to flow through to your Delta sink, you'll need to add mergeSchema=true to your writeStream options — but that alone isn't enough. Structured Streaming needs to be restarted to pick up the schema change. Build this into your ops runbook: schema change → update checkpoint schema handling → rolling restart. Catching this late in an incident is painful.

Lakeflow Declarative Pipelines ● GA
Formerly Delta Live Tables (DLT) · Define what, let Spark figure out how
Overview
Code Patterns
Tips

Delta Live Tables — now rebranded as Lakeflow Spark Declarative Pipelines — flips the pipeline authoring model on its head. Instead of writing imperative code that says "read this, transform it, write it there, in this order", you write declarative table definitions using @dlt.table decorators and Databricks figures out execution order, dependency resolution, and retry logic automatically. You describe the what, Spark handles the how.

The most impactful practical feature is data quality expectations. You annotate your table function with rules like @dlt.expect_or_drop("valid_order_id", "order_id IS NOT NULL") and Databricks tracks expectation pass/fail rates in the pipeline event log. You can alert on quality degradation, quarantine bad rows, or fail the entire pipeline — without writing any custom validation logic. This is the thing that makes medallion architecture actually enforceable rather than aspirational.

The big 2025 news: Databricks donated Spark Declarative Pipelines to the Apache Spark project as an open standard. This means your pipeline code is no longer locked to Databricks — it runs on any Spark deployment. For existing DLT users, nothing changes; it's 100% backwards-compatible. You just get a more powerful, open foundation going forward.

# Lakeflow Declarative Pipeline — Medallion Architecture
import dlt
from pyspark.sql.functions import col, current_timestamp

# Bronze: raw ingest from Auto Loader
@dlt.table(
  name="bronze_orders",
  comment="Raw orders from ADLS landing zone"
)
def bronze_orders():
  return (spark.readStream
    .format("cloudFiles")
    .option("cloudFiles.format", "json")
    .load("/mnt/landing/orders")
    .withColumn("ingested_at", current_timestamp())
  )

# Silver: cleaned + validated
@dlt.table(name="silver_orders")
@dlt.expect_or_drop("valid_order_id", "order_id IS NOT NULL")
@dlt.expect_or_drop("valid_amount", "amount > 0")
def silver_orders():
  return (dlt.read_stream("bronze_orders")
    .select("order_id", "customer_id", 
            col("amount").cast("double"), "ingested_at")
  )

# Gold: business aggregate
@dlt.table(name="gold_daily_revenue")
def gold_daily_revenue():
  return (dlt.read("silver_orders")
    .groupBy("customer_id")
    .agg({"amount": "sum"})
  )

Interview questions on DAB range from basic ("what is it") to architectural ("how does it fit with Terraform"). Know all of these.

Q1 — Easy What is a Databricks Asset Bundle and how is it different from deploying via the UI or ARM templates? ▼

A DAB is a project-level YAML definition of all Databricks resources — jobs, DLT pipelines, clusters, permissions — that can be deployed to any workspace via a single CLI command. Unlike UI-based deployment (ClickOps, no auditability, no version control) and ARM templates (Azure-specific, not Databricks-native, complex to maintain), DAB is:

1. Git-native — lives in the repo alongside the code it deploys
2. Multi-environment aware — one config, multiple targets with variable overrides
3. Databricks-native — understands DLT pipelines, cross-resource references, UC permissions
4. Auditable — every config change is a Git commit, reviewable as a PR

Interview angle: "It replaces the ADF ARM template export pattern I used before — instead of exporting a JSON blob that's hard to diff and review, every pipeline definition is clean YAML that any engineer can read."
Q2 — Easy What does mode: development do and why does it matter? ▼

Development mode does three things automatically: (1) Prefixes every resource name with your username — so your pipeline is called mohit.upadhyay-etihad-flight-dlt-dev instead of just etihad-flight-dlt-dev. This prevents resource name collision when multiple engineers deploy simultaneously. (2) Attaches a dev tag to all resources for cost attribution and easy cleanup. (3) Does NOT pause scheduled jobs — be careful, schedules still run in dev mode unless you explicitly set pause_status: PAUSED.

Common gotcha: dev mode prefix only works within the same workspace. If you accidentally deploy a dev bundle to the prod workspace, the username prefix won't protect you — the resource still gets created in prod.
Q3 — Medium How do you deploy the same DLT pipeline to three different Unity Catalog environments without duplicating YAML? ▼

Use DAB's variables + targets pattern. Define the pipeline resource once using variable references (${var.catalog}, ${bundle.target}). In the targets: block, each environment overrides only the values that differ — catalog name, ADLS account, cluster size, pipeline channel. The pipeline definition itself is written once.

pipelines:
  etihad_flight_dlt:
    name: etihad-dlt-${bundle.target}      # etihad-dlt-dev/qa/prod
    catalog: ${var.catalog}                 # etihad_dev/qa/prod
    channel: ${var.pipeline_channel}        # PREVIEW/PREVIEW/CURRENT
Deploy: databricks bundle deploy -t dev → -t qa → -t prod. The same YAML, three workspaces, fully isolated.
Q4 — Medium A job needs to trigger the DLT pipeline it depends on. How do you wire this in DAB without hardcoding pipeline IDs? ▼

Use a DAB cross-resource reference. In the job's pipeline_task, set pipeline_id: ${resources.pipelines.etihad_flight_dlt.id}. DAB deploys the pipeline first, captures its assigned ID, then substitutes it into the job definition before deploying the job. You never touch environment-specific IDs manually.

This is the equivalent of what you'd need a complex ADF parameter passing chain to achieve — DAB handles it declaratively in one line.

The cross-reference also enforces deployment order: DAB deploys the pipeline before the job, because the job's config depends on the pipeline's ID. Dependency-ordered deployment is free.
Q5 — Medium How do you set up DAB CI/CD in Azure DevOps with a manual approval gate for production? ▼

Three-stage ADO pipeline: (1) Validate — runs on every PR, calls databricks bundle validate -t dev. Catches YAML errors before anything touches cloud. (2) DeployQA — runs on merge to main with condition Build.Reason = IndividualCI, deploys to QA automatically. (3) DeployProd — uses ADO's deployment job type with environment: etihad-prod. Configure approval checks in the ADO Environment settings — the stage pauses until an approver clicks Approve.

Secrets live in ADO Library Variable Groups, not in YAML. Separate PATs per environment, injected as DATABRICKS_TOKEN environment variable at runtime.

Key interview point: the ADO Environment approval is NOT a pipeline condition — it's a separate governance gate configured in ADO's Environments UI. This is what makes it a proper change management control, not just a YAML if-condition.
Q6 — Hard How does DAB relate to Terraform in a production Databricks platform? When would you use each? ▼

They operate at different abstraction levels and are complementary, not competing:

Terraform manages platform infrastructure: workspaces, Unity Catalog catalogs and schemas, network configuration (VNet injection, private endpoints), storage accounts, service principals, IP access lists. Terraform is the right tool because these are long-lived, infrequently-changed resources that multiple teams depend on.

DAB manages workload resources: jobs, DLT pipelines, cluster configs, notebook references, permissions on specific resources. DAB is the right tool because these change with every code release and are owned by a specific pipeline/team.

The interface: Terraform creates the catalog. DAB deploys the pipeline that writes to it. Terraform outputs (catalog name, workspace URL) feed into DAB variables. This creates a clean separation: infra changes go through Terraform review, workload changes go through DAB CI/CD.

Staff-level answer: "For Etihad, Terraform manages the three workspaces and UC catalog hierarchy. DAB manages the 8 pipelines and 12 jobs that run in them. The Terraform outputs are stored in Azure DevOps variable groups and consumed by DAB's target workspace configs."
Q7 — Hard You renamed a resource key in DAB and deployed to prod. Now you have two versions of the resource running. How do you fix this? ▼

DAB tracks resources by their YAML key. When you rename a key, DAB treats the old key as deleted and the new key as a new resource — so it creates the new one but does NOT automatically delete the old one. In prod, this means both the old pipeline (running on its schedule) and the new one are active simultaneously.

Immediate fix: Go to the Databricks UI → Jobs/Pipelines, pause or delete the old resource manually. Then verify only the new one is running.

Clean fix via DAB: Before renaming, run databricks bundle destroy -t prod to remove all managed resources, then rename, then bundle deploy -t prod. Note: destroy-then-deploy causes brief downtime — schedule it in a maintenance window.

Prevention: Always bundle validate and review the "resources to be created/deleted" diff before deploying. Never rename keys without a destroy step.

This is the DAB equivalent of the Terraform "resource drift" problem — Terraform has the same issue with resource renames. If you've used Terraform, this pattern is familiar.

💡 The safest backfill

Isolated · Validated · Idempotent · Reversible. All four properties must hold. Miss one and you're one retry away from a production incident.

1
Define exact scope — before touching any data
Be painfully clear: which date range, which partitions, which tables, which downstream consumers. Most backfill disasters start because scope was vague.
# Scope checklist — fill this before writing a single line of transform code
scope = {
    "table"        : "silver.transactions",
    "start_dt"     : "2026-01-01",
    "end_dt"       : "2026-03-31",
    "partitions"   : ["month=01", "month=02", "month=03"],
    "root_cause"   : "MERGE on txn_id alone — currency_code ignored",
    "consumers"    : ["gold.revenue_summary", "pbi.fx_dashboard"],
    "approved_by"  : "Data Engineering Lead",
}
2
Reprocess in isolation — never write to prod first
Use a shadow table, temp location, or staging dataset. This gives you room to validate before users see anything. DEEP CLONE creates an independent Delta copy with its own transaction log.
# DEEP CLONE = independent copy, full Delta log, zero data copy cost
spark.sql("""
    CREATE TABLE silver.transactions_bf_v2
    DEEP CLONE silver.transactions
    LOCATION '/mnt/adls/backfill/transactions_bf_v2'
""")

# Verify clone integrity before proceeding
orig  = spark.table("silver.transactions").count()
clone = spark.table("silver.transactions_bf_v2").count()
assert orig == clone, f"Clone mismatch: {orig} vs {clone}"
3
Make writes idempotent — MERGE, not INSERT
If the job runs twice, the final result must be identical. That means: merge instead of blind insert, dedupe by business key, use partition-aware writes, track backfill run IDs. Retries must not create a second incident.
# Composite key MERGE — safe to rerun any number of times
spark.sql("""
    MERGE INTO silver.transactions_bf_v2  AS tgt
    USING      silver.transactions_staged AS src
        ON  tgt.txn_id         = src.txn_id
        AND tgt.currency_code  = src.currency_code
        AND tgt.partition_date = src.partition_date

    WHEN MATCHED THEN UPDATE SET
        tgt.amount_inr      = src.amount_usd * src.fx_rate,
        tgt.fx_rate         = src.fx_rate,
        tgt.backfill_run_id = src.run_id,
        tgt.updated_at      = current_timestamp()

    WHEN NOT MATCHED THEN INSERT *
""")
🚨 Never do this

INSERT INTO silver.transactions_bf_v2 SELECT * FROM staged — blind insert with no key check. A second run (retry, re-trigger, accident) doubles every row. Silent. Devastating.

4
Validate before publishing — do not trust a green job
A successful Spark job does not mean correct data. Validate row counts, null rates, duplicates, key metrics, partition coverage, and downstream dashboard sanity. This is the gate that protects prod.
# Validation suite — run against shadow table before any swap
orig  = spark.table("silver.transactions")
fixed = spark.table("silver.transactions_bf_v2")

# 1. Row count parity
assert fixed.count() == orig.count(), "Row count mismatch!"

# 2. Zero nulls in critical columns
nulls = fixed.filter("amount_inr IS NULL OR fx_rate IS NULL").count()
assert nulls == 0, f"{nulls} null rows — abort"

# 3. Metric ratio within expected shift range
old_sum = orig.agg({"amount_inr": "sum"}).collect()[0][0]
new_sum = fixed.agg({"amount_inr": "sum"}).collect()[0][0]
ratio   = new_sum / old_sum
assert 1.02 < ratio < 1.08, f"Ratio {ratio:.3f} outside safe range"

# 4. No composite key duplicates
dupes = (fixed
    .groupBy("txn_id", "currency_code", "partition_date")
    .count()
    .filter("count > 1")
    .count()
)
assert dupes == 0, f"{dupes} duplicate composite keys!"
5
Swap atomically — users must never see half-old / half-new data
Use Delta's replaceWhere to overwrite only the affected partitions in a single atomic operation. Users see either all-old or all-new — never a mix. Always keep a rollback path.
# Atomic partition swap — touches only the affected date range
(spark.table("silver.transactions_bf_v2")
    .filter("partition_date BETWEEN '2026-01-01' AND '2026-03-31'")
    .write
    .format("delta")
    .mode("overwrite")
    .option("replaceWhere",
            "partition_date BETWEEN '2026-01-01' AND '2026-03-31'")
    .saveAsTable("silver.transactions")
)

# Rollback if something looks wrong post-swap:
# RESTORE TABLE silver.transactions VERSION AS OF <n>
6

Backfills are not only technical work. They affect trust. Tell stakeholders what changed, why it changed, which dates were affected, which metrics will move, and who approved the correction.

Your MERGE job ran last night and revenue in the dashboard jumped 15%. This morning you realise it's wrong — the MERGE used a partial key. How do you recover without downtime?
Hard▼
Lead with the principle: "I treat this as a production rollback, not a debug session. First thing is to assess impact — which tables, which partitions, which consumers."

Recovery steps:
1. Immediately roll back the affected table: RESTORE TABLE silver.transactions VERSION AS OF <n-1> — this is instant, atomic, and brings dashboards back to pre-MERGE state
2. Alert stakeholders that the jump was a data error and the table is being restored
3. Identify root cause — in this case, MERGE matched on partial key, wrong rows updated
4. Build the fix: DEEP CLONE to shadow, correct the MERGE with composite key, validate metric ratio in expected range
5. Once shadow passes all validation checks, swap atomically with replaceWhere
6. Notify stakeholders of the corrected figure and explain the delta

Key point to add: "Delta Lake's RESTORE is the get-out-of-jail-free card here. It's why I always recommend teams keep a retention policy of at least 30 days on Delta history for production tables — 7-day default is too short for business-critical incident response."
What is the difference between an idempotent write and a safe backfill? Can you have one without the other?
Medium▼
Idempotent write — an operation that produces the same result regardless of how many times it runs. MERGE with a composite business key is idempotent: running it twice gives the same output. Blind INSERT is not: running it twice doubles every row.

Safe backfill — a broader operational pattern: scoped, isolated, idempotent, validated, atomically swapped, communicated. Idempotent writes are one required component.

Can you have one without the other? Yes — you can write idempotent code (MERGE) but still have an unsafe backfill because you write directly to prod (no isolation), skip validation, and don't have a rollback path. The idempotency only protects you from the "ran twice" failure mode — not from writing wrong data, corrupting live partitions, or breaking downstream consumers.

Interview punchline: "Idempotency is necessary but not sufficient. A safe backfill needs all four properties: isolated, idempotent, validated, reversible."
How would you handle a backfill on a streaming table where the live pipeline is still running and writing new data?
Hard▼
This is the trickiest backfill scenario — the live stream is continuously appending while you're trying to fix historical partitions.

Key insight: Use partition-scoped isolation. The live stream writes to current partitions (today, this hour). The backfill only touches historical partitions (last month). As long as you scope the replaceWhere to only the historical date range, there's no conflict.

Steps:
1. Define the exact partition boundary: backfill scope must not overlap with the live stream's write window
2. DEEP CLONE only the historical partitions to shadow — don't clone the whole table
3. Run corrections on shadow
4. Validate — pay special attention to the boundary partition (the month/day where live and historical meet)
5. Swap with replaceWhere scoped strictly to historical range — live stream is unaffected

What to avoid: TRUNCATE + INSERT on the full table while the stream runs — this creates a race condition and you'll mix old and new data in the boundary partition. Always use partition-scoped replaceWhere, never full-table overwrite while a stream is running.
When would you NOT use replaceWhere for the swap step? What are the alternatives?
Medium▼
When replaceWhere doesn't work:

1. No partition column — replaceWhere requires at least one partition predicate. If your table isn't partitioned, you can't use it. Use a full table MERGE instead, or rename pattern (shadow → prod via atomic table rename).

2. You need to backfill across all partitions — if the error affects every partition, a full table swap is cleaner than running replaceWhere for every partition separately. Pattern: write corrected shadow table → ALTER TABLE RENAME (zero-downtime if consumers use the same name).

3. Liquid clustering tables — replaceWhere doesn't work well with Liquid Clustering tables because they don't have discrete partition columns. Use MERGE or a full overwrite with SHALLOW CLONE + RENAME pattern instead.

Rename pattern (zero-downtime):
ALTER TABLE silver.transactions RENAME TO silver.transactions_old
ALTER TABLE silver.transactions_bf_v2 RENAME TO silver.transactions
DROP TABLE IF EXISTS silver.transactions_old

One atomic operation. Downstream consumers using the table name see no interruption.

Liquid Clustering ● GA
Partition-free physical data layout · Replaces Hive partitioning + Z-Ordering
Overview
How to Use
Internals
Tips & Interview Q

Liquid Clustering is Delta Lake's replacement for both Hive-style partitioning and Z-Ordering. The core idea: instead of requiring you to pick a partition column upfront and live with that choice forever, Liquid Clustering lets you define clustering keys on any column(s) and then incrementally re-clusters only the files that need it — without ever rewriting the entire table.

The "liquid" name refers to the fact that the clustering layout is flexible and can evolve. You can change clustering keys on an existing table with a single ALTER TABLE command and the next CLUSTER BY run will gradually migrate the layout. This was simply impossible with partitioning — changing a partition column meant a full table rewrite.

Under the hood, Databricks uses a cost-based optimiser that examines your query patterns and the current file layout. When you run OPTIMIZE (manually or via Predictive Optimization), it computes which files are "poorly clustered" for your declared keys and rewrites only those, leaving well-clustered files alone. This is incremental by design.

✦ Why it matters for interviews

Interviewers frequently ask "when would you choose Liquid Clustering over partitioning?" The answer: when cardinality is high (user IDs, transaction IDs — these make terrible partition columns), when you need to cluster on multiple columns simultaneously, when your data skews heavily, or when you anticipate needing to change the clustering strategy later. Partitioning is still valid for very-low-cardinality, time-based splits used in data retention policies.

Creating a table with Liquid Clustering

-- CREATE with clustering keys
CREATE TABLE sales (
  order_id     BIGINT,
  customer_id  BIGINT,
  order_date   DATE,
  amount       DECIMAL(12,2),
  region       STRING
)
CLUSTER BY (customer_id, order_date);   -- multi-column clustering

-- OPTIMIZE triggers incremental clustering
OPTIMIZE sales;

-- Change clustering keys on existing table (no rewrite yet)
ALTER TABLE sales CLUSTER BY (region, order_date);
OPTIMIZE sales;  -- gradually migrates layout to new keys

-- PySpark equivalent
spark.sql("""
  CREATE TABLE silver.sales
  CLUSTER BY (customer_id, order_date)
  AS SELECT * FROM bronze.raw_sales
""")

-- Check current clustering config
DESCRIBE DETAIL sales;
-- Look at: clusteringColumns field in output
  • 1
    Choose your clustering keys
    Pick columns your queries most commonly filter on. High-cardinality columns (customer_id, transaction_id) are ideal — these made terrible partition columns but work perfectly as clustering keys.
  • 2
    Declare at CREATE or ALTER TABLE
    Use CLUSTER BY (col1, col2) in your CREATE TABLE. For existing tables use ALTER TABLE tbl CLUSTER BY (cols) — no immediate rewrite occurs.
  • 3
    Run OPTIMIZE (or use Predictive Optimization)
    Manually OPTIMIZE table_name or let Predictive Optimization do it. Only files that need reclustering are touched — first run may be slow, subsequent runs are fast.
  • 4
    Validate with query plans
    Run a filtered query and check EXPLAIN — you should see file skipping stats confirming fewer files were scanned.

How it works under the hood

Liquid Clustering is built on Delta Lake's transaction log and the Hilbert curve space-filling algorithm. When OPTIMIZE runs on a clustered table:

  • It reads the _delta_log statistics to find which files have high overlap on the clustering key range — these are "poorly clustered" files.
  • It co-locates rows with similar clustering key values using a Hilbert curve ordering. This is superior to Z-ordering because it has better locality preservation — adjacent values in N-dimensional space map to adjacent positions on the curve with fewer gaps.
  • Only files above a "clustering score" threshold are rewritten. Files already well-clustered are untouched — this is the incremental property.
  • Min/max statistics in the transaction log are updated after clustering — this is what enables file skipping at query time.

Interview scenarios

💬 "When would you not use Liquid Clustering?"

When your table is tiny (no point optimizing a 1GB table). When you need physical partition pruning for data retention policies — e.g., ALTER TABLE DROP PARTITION (year=2020) to delete old data instantly without a rewrite. Liquid Clustering has no equivalent fast-delete by partition. Also, when your downstream tools (e.g., Spark on EMR) don't support Delta's clustering stats — they won't benefit from file skipping.

💬 "How does file skipping work with Liquid Clustering?"

Delta Lake maintains per-file min/max statistics for each column in the transaction log. After clustering, rows with similar customer_id values are colocated in the same files. When you query WHERE customer_id = 12345, the query engine checks the stats and skips any file whose customer_id min/max range doesn't include 12345. The more clustered the data, the fewer files are scanned.

  • Max recommended clustering keys: 2–4 columns. More columns = diminishing returns and slower OPTIMIZE.
  • Liquid Clustering is particularly powerful for MERGE operations — the merge predicate columns should match your clustering keys so fewer files need to be rewritten per merge.
  • Run DESCRIBE DETAIL table_name and check numFilesOptimized and clusteringColumns to validate the layout.
  • Liquid Clustering works with both managed and external Delta tables on Unity Catalog.

Predictive Optimization is Databricks' answer to a very common pain point: teams forget to run OPTIMIZE and VACUUM, or run them on a fixed schedule that doesn't match actual data arrival patterns. This wastes both compute (running OPTIMIZE on tables that haven't changed) and storage (VACUUM running too infrequently leaves too many old files around).

It's a Unity Catalog-only feature. Tables not registered in Unity Catalog (Hive Metastore tables) are not eligible. The optimization runs as a background serverless job — it doesn't consume your cluster compute, it uses Databricks-managed serverless capacity, so there's no cost for the compute — only for the storage operations themselves.

What the ML model actually looks at

Predictive Optimization's model tracks signals across three dimensions to decide when to act:

  • Write pressure: How frequently is new data being appended or merged? High-frequency streaming writes create many small files faster — OPTIMIZE is triggered more aggressively. A table written once a week needs far less intervention.

  • Historical query activity: If a table is frequently queried, optimization has higher ROI — poorly-clustered files cost more in aggregate read time. Dormant tables may never get optimized even if their layout is suboptimal.

RocksDB State Store ● GA
SSD-backed state for Spark Structured Streaming · Replaces in-memory HashMapStateStore
Overview
Enable & Configure
Internals
Tips & Interview Q

When Spark Structured Streaming performs stateful operations — aggregations over time windows, stream-stream joins, mapGroupsWithState, flatMapGroupsWithState — it needs to store state for each key between micro-batches. The original implementation kept all of this in JVM heap memory as a HashMap. This works fine for small state, but breaks down fast when state grows large: executor OOM errors, long GC pauses, and query failures.

RocksDB State Store replaces that in-memory map with an embedded key-value store backed by SSD (local NVMe on Databricks clusters). RocksDB is a log-structured merge tree (LSM) database created by Facebook, designed for high-throughput, low-latency key-value operations on solid-state storage. The result: your streaming state can grow to terabytes without hitting memory limits, and point lookups are sub-millisecond.

This is why you see it mentioned in the context of sessionization, real-time fraud detection, and IoT device tracking — any use case where you maintain per-key state over long time windows and can't afford to keep everything in RAM.

SSD-backed State
State lives on local NVMe, not JVM heap — eliminates OOM for large state workloads
⚡
Sub-millisecond Lookups
RocksDB's LSM-tree gives <1ms point reads even at scale — critical for low-latency streaming
📦
Checkpointing to Cloud
State is checkpointed to ADLS / S3 for fault tolerance — survives cluster restarts

Changelog checkpointing — the key performance win

By default, Spark snapshots the entire state store at each checkpoint. For large state, this means copying gigabytes of RocksDB files to cloud storage every micro-batch — expensive and slow. Changelog checkpointing writes only the changes (new/updated/deleted keys) since the last checkpoint — analogous to how Delta Lake's transaction log works. Enable with spark.databricks.streaming.rocksdb.changelogCheckpointing.enabled = true.

💬 "When would you choose RocksDB over the default state store?"

Any time your stateful streaming query has: (1) a large key space — millions of distinct session_id, customer_id, or device_id keys; (2) long time windows — 1-hour or 24-hour windows mean state persists much longer and grows large; (3) complex state per key — storing full session event histories per user, not just a count. If your state is small (thousands of keys, short windows), the default HashMapStateStore is faster since it's pure memory with no I/O overhead.

💬 "What's the relationship between RocksDB and Auto Loader checkpointing?"

These are two different uses of RocksDB. Auto Loader uses RocksDB internally to track which files it has already processed — that's an implementation detail you don't configure. The RocksDB State Store is a separate feature you explicitly enable for stateful streaming operations (aggregations, joins, mapGroupsWithState). They both use RocksDB under the hood but serve completely different purposes.

  • Always enable changelogCheckpointing alongside RocksDB — snapshot-based checkpointing negates much of the performance benefit for large state.
  • Tune memoryBudgetMB based on your average state size per partition. Too low → excessive SSD I/O. Too high → OOM from other executor memory consumers.
  • RocksDB state is lost if the executor's local disk fails. The checkpoint in cloud storage is the recovery source — ensure your checkpoint path is reliable (ADLS Gen2 / S3 with versioning).
  • Monitor state size using Spark UI → Streaming tab → State Operators metrics. Growing unbounded state with no watermark is the #1 cause of RocksDB-backed query slowdown.

Together, CDF + AutoCDC form the recommended pattern for propagating changes through Medallion layers — Bronze captures raw changes from source systems, Silver applies those changes with AutoCDC to maintain a clean current state, Gold aggregates from Silver.

AutoCDC with APPLY CHANGES (in DLT / Declarative Pipelines)

-- SQL syntax in Declarative Pipelines (DLT)
CREATE OR REFRESH STREAMING TABLE silver.customers;

APPLY CHANGES INTO silver.customers
FROM STREAM(bronze.customers_cdc)
KEYS (customer_id)                       -- primary key columns
APPLY AS DELETE WHEN operation = 'DELETE'
APPLY AS TRUNCATE WHEN operation = 'TRUNCATE'
SEQUENCE BY updated_at                   -- handle out-of-order events
COLUMNS * EXCEPT (operation, updated_at); -- exclude CDC metadata cols
# Python API — same in DLT notebooks
import dlt

dlt.create_streaming_table("silver_customers")

  target = "silver_customers",
  source = "bronze_customers_cdc",
  keys = ["customer_id"],
  sequence_by = col("updated_at"),      # deduplication ordering
  apply_as_deletes = expr("operation = 'DELETE'"),
  except_column_list = ["operation", "updated_at"],
  stored_as_scd_type = "2"               # SCD Type 2 history tracking
)

How CDF works under the hood

When enableChangeDataFeed = true, Delta Lake writes additional change data files alongside normal data files whenever DML operations occur. Specifically:

  • For UPDATE and DELETE operations, Delta writes a _change_data directory inside the table path containing Parquet files with the before and after images of affected rows. These are stored separately from the main data files.
  • For INSERT operations, no extra files are written — the change records are derived from the normal data files since all inserted rows are "new."
  • The _delta_log transaction log entry for each commit includes a pointer to the CDF change files, so CDF readers can reconstruct the exact set of changes for any version range.

✦ SCD Type 2 with AutoCDC — what it generates

With stored_as_scd_type = "2", AutoCDC automatically adds __START_AT and __END_AT columns to your target table using the sequence column values. Current rows have __END_AT = NULL. Historical rows show the period they were valid. You don't write any of that logic — AutoCDC generates it from the sequence ordering. This completely replaces the complex SCD2 MERGE patterns you'd otherwise write by hand.

💬 "What's the difference between CDF and AutoCDC?"

CDF (Change Data Feed) is the low-level Delta feature — it tracks row-level changes and lets you query them as _change_type records. It's the source of truth for what changed. AutoCDC (APPLY CHANGES INTO) is the higher-level DLT feature that consumes those changes and applies them to a target Delta table declaratively, handling deduplication, ordering, SCD1/2 logic, and deletes automatically. CDF is plumbing; AutoCDC is the smart appliance built on top of it.

💬 "What are the gotchas with enabling CDF?"

Three main ones: (1) Storage overhead — CDF writes extra change files for every UPDATE and DELETE. High-churn tables with frequent updates can see 20–40% storage increase. (2) Cannot be disabled retroactively without losing history — once enabled, the CDF files are part of the table. Disabling CDF stops writing new change files but old ones remain and count toward VACUUM. (3) VACUUM interactions — CDF files follow the same retention window as regular Delta files. Vacuum them too aggressively and you lose the ability to read old change ranges.

⚠ AutoCDC is DLT-only

The APPLY CHANGES INTO / apply_changes() API only works inside Lakeflow Declarative Pipelines (DLT). You cannot call it from a regular notebook or Spark job. If you need CDC application logic outside DLT, you must implement it manually with MERGE INTO and handle deduplication and ordering yourself.

  • The SEQUENCE BY column is critical — it determines which version of a row "wins" when out-of-order events arrive. Always use an application-level timestamp or monotonically increasing sequence number, not the Kafka offset (which can reset).
  • For SCD Type 2, query current state with WHERE __END_AT IS NULL — that filter is added automatically by AutoCDC and is optimized via Delta statistics.
  • CDF reads are versioned — you can always recover from a specific startingVersion, making CDF a reliable foundation for incremental pipelines that need to restart from a known point.
  • In Healthcare and Finance — two of your client domains — CDF + SCD Type 2 is the standard pattern for maintaining historical patient/account records with full audit trails. This is a strong interview talking point.

Tuning & Verification

Before AQE (Spark < 3.0), the query planner made all decisions upfront based on table statistics — which were often stale, missing, or just wrong. The plan was fixed at compile time and executed blindly regardless of what the actual data looked like.

AQE flips this. It re-examines the query plan after each shuffle stage completes, using real runtime statistics, and re-optimizes the remaining plan on the fly.

💡 In Databricks

AQE is on by default on all clusters running DBR 7.3+. You do not need to set it manually in most cases. However knowing the individual knobs — especially skew thresholds — is what separates a senior who can tune it from a junior who just knows it exists.

-- Enable (already on in Databricks, useful for local Spark)
SET spark.sql.adaptive.enabled = true;
SET spark.sql.adaptive.coalescePartitions.enabled = true;
SET spark.sql.adaptive.skewJoin.enabled = true;

Optimization 1 — Dynamic Partition Coalescing

Problem: spark.sql.shuffle.partitions = 200 (default). After a heavy filter removes 95% of data, you have 200 near-empty shuffle partitions — 200 tasks doing almost nothing, running sequentially.

What AQE does: After the shuffle completes, it reads actual partition sizes and merges adjacent small partitions into larger ones — without re-shuffling the data.

-- Target size after coalescing (default 64MB, bump to 128MB for large jobs)
SET spark.sql.adaptive.advisoryPartitionSizeInBytes = 128MB;

Optimization 2 — Dynamic Join Strategy Switching

Problem: Optimizer estimated Table B at 500MB → chose Sort-Merge Join. At runtime, after filter pushdown, Table B is actually 8MB. SMJ is the wrong choice but the static plan is locked in — two full shuffles wasted.

What AQE does: After shuffling the build side, AQE checks its actual size. If it fits under the broadcast threshold, it converts SMJ → Broadcast Hash Join on the fly — eliminating the sort and the second shuffle entirely.

-- AQE-specific broadcast threshold (separate from static autoBroadcastJoinThreshold)
SET spark.sql.adaptive.autoBroadcastJoinThreshold = 30MB;

-- Interview tip: don't always hardcode broadcast() hints
-- A wrong hint PREVENTS AQE from choosing a better strategy
-- Let AQE decide unless you have a specific reason to override

Optimization 3 — Skew Join Optimization

Problem: Joining orders (1B rows) with customers on customer_id. One enterprise account has 40M orders — one partition is 8GB while all others are ~200MB. Every executor finishes in 2 min. That one straggler runs 45 min. Your entire stage is blocked.

What AQE does: Detects skewed partitions after the shuffle using two thresholds. Splits them into sub-partitions and duplicates the matching partition from the other table to join against each split — all running in parallel on separate executors.

-- Partition flagged as skewed when BOTH conditions are true:
SET spark.sql.adaptive.skewJoin.skewedPartitionFactor = 5;
-- size > 5x the median partition size
SET spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes = 256MB;
-- AND size > 256MB absolute

-- Tune down factor to 3 if stragglers persist (more aggressive detection)
SET spark.sql.adaptive.skewJoin.skewedPartitionFactor = 3;

This is where interviewers separate strong candidates. AQE is powerful — but it has hard limits that matter in production.

🔴 Skew in the join key value itself

If 60% of your rows have customer_id = NULL, AQE detects the giant partition and splits it — but you're still joining millions of nulls against millions of nulls. AQE can split the partition but can't fix the underlying data problem. Fix this upstream: filter nulls before the join or coalesce them to a sentinel value.

🔴 First shuffle stage runs blind

AQE only kicks in after a shuffle completes. The very first stage of a query executes with the static plan — no runtime stats available yet. If your source scan is massively skewed (e.g., one Parquet file is 50GB, others are 200MB), AQE can't help. Fix: use OPTIMIZE + Liquid Clustering on the source table to even out file sizes before the query runs.

⚠ Python UDFs break the optimization chain

AQE re-optimizes between shuffle stages. A Python UDF creates an opaque JVM→Python boundary — Spark cannot see through it to inspect the data or apply further optimizations. The optimization chain stops at the UDF. Keep transformations in native Spark SQL / DataFrame API wherever possible. If you must use a UDF, isolate it to the last stage of the pipeline.

⚠ Window functions are not covered by skew handling

AQE's skew join optimization applies to joins — not to window function shuffles. If your partitionBy("customer_id") key is skewed, AQE does nothing. Fix manually: salt the key, pre-aggregate, or repartition before the window operation. This is a very common production gotcha on large dimension tables.

⚠ Broadcast threshold is size-based, not memory-relative

AQE won't broadcast a 2GB table even if your executors have 64GB memory each. The threshold is an absolute size check. Tune autoBroadcastJoinThreshold upward if your executors have headroom — don't leave it at the default 10MB if your cluster can handle larger broadcasts.

💡 Summary: what AQE cannot fix
Null skew in join key → fix upstream with filter/sentinel
First stage (no shuffle yet) → fix with OPTIMIZE on source table
Python UDF boundary → use native Spark SQL instead
Window function skew → salt key or pre-aggregate manually
Broadcast threshold too low → tune autoBroadcastJoinThreshold

Context: Daily job aggregates sales for the current month only from a 3-year, 500M-row fact table. Filter removes ~92% of data. With shuffle.partitions = 200, you get 200 near-empty partitions after the filter — job takes 18 min when it should take 3.

The Mechanics

Structured Streaming tracks processed offsets in a checkpoint directory. Normally this log only moves forward. Streaming Rewind lets you set the stream's read position backward to an earlier offset or timestamp, while keeping the existing state store intact.

Triggering Streaming Rewind — DBR 18.2+

The exact API surface is still in Beta. Based on confirmed release notes, the pattern follows the SDP checkpoint reset approach, which can be triggered programmatically:

# Pattern 1 — Rewind via SDP (Lakeflow Declarative Pipelines)
# Identify the flow writing to your silver table
from databricks.sdk import WorkspaceClient
from databricks.sdk.service.pipelines import ResetRequest

w = WorkspaceClient()

# Reset the checkpoint of a specific streaming flow to a timestamp
# This is the rewind — Spark will replay from this point on next run
w.pipelines.reset(
    pipeline_id="your-pipeline-id",
    full_refresh=False,          # do NOT full refresh — preserves state
    full_refresh_selection=["silver_events"]  # only reset this table's flow
)
# Pattern 2 — Raw Structured Streaming with startingTimestamp
# Stop the stream, then restart from a target timestamp

# Step 1: Stop running stream
query.stop()

# Step 2: Restart with startingTimestamp to rewind source
# The state store at checkpointLocation is preserved
df = spark.readStream     .format("delta")     .option("startingTimestamp", "2024-01-15T13:55:00") \  # 1:55PM
    .table("bronze_events")

query = df     .transform(apply_silver_logic)     .writeStream     .format("delta")     .outputMode("append")     .option("checkpointLocation", checkpoint_path) \  # SAME checkpoint = state preserved
    .option("mergeSchema", "true")     .start("silver_events")
# Step 3: Reconcile sink — MERGE to overwrite bad records
# Records between 2PM–3PM were already written — fix them
spark.sql("""
    MERGE INTO silver_events AS target
    USING (
        SELECT * FROM silver_events_replay
        WHERE event_time BETWEEN '2024-01-15T14:00:00' AND '2024-01-15T15:00:00'
    ) AS source
    ON target.event_id = source.event_id
    WHEN MATCHED THEN UPDATE SET *
    WHEN NOT MATCHED THEN INSERT *
""")
💡 Source Retention Check Before You Rewind

Verify your source still has the data before attempting rewind. For Kafka/Event Hubs, check retention period. For Delta CDF, check DESCRIBE HISTORY bronze_events — you need the version that corresponds to your 1:55PM target to still exist (not yet VACUUMed).

What is the difference between Streaming Rewind and a full state reset? Why does one preserve state and the other doesn't?
Medium▼
Full state reset deletes the entire checkpoint directory — both the offset log AND the state store. The offset log tracks what source data has been processed. The state store holds intermediate results (window aggregations, join state, watermark). Deleting both means Spark starts completely from scratch.

Streaming Rewind only modifies the offset log — it sets the read position back to an earlier point. The state store directory is untouched. When the stream restarts, it reads source data from the rewind point but continues accumulating state on top of what was already in the store.

The nuance: replaying data against existing state can produce unexpected aggregation results if you have stateful operations. For purely stateless transforms (filter, project, map), rewind is clean. For stateful operations (window aggregations, streaming joins), you need to reason carefully about what replaying older events does to existing state.
When would you NOT use Streaming Rewind even in DBR 18.2?
Medium▼
Three situations where Rewind is not the right tool:

1. Source retention expired — Kafka default retention is 7 days. Event Hubs default is 1 day. If you need to rewind 2 weeks and your source has already purged those offsets, rewind can't help. Fall back to reprocess from raw (Bronze layer).

2. Stateful pipelines with complex window logic — Replaying older events into an existing state store that has already closed those windows can lead to incorrect results. The watermark has moved forward — late events may be dropped. Full reset may be safer if state correctness is critical.

3. Sink doesn't support idempotent writes — If your downstream is a non-Delta sink (external DB, REST API, queue), replaying records will create duplicates with no clean way to reconcile. Full reset or reprocess from raw gives you a clean slate.

Interview punchline: "Rewind is the best default choice in DBR 18.2, but like any recovery tool, you apply it after verifying the three prerequisites: source retention, stateless or carefully-understood stateful logic, and an idempotent sink."
